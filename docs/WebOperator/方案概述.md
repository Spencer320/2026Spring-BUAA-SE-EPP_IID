# Web Operator (网页操作工具) 升级方案概述

## 1. 背景与目标

当前系统使用 Tavily 作为通用搜索引擎，虽然在通用问答和 RAG 场景下表现良好，但无法满足深度学术检索和特定网站复杂交互的需求。为了让 AI 具备真正的“网页操作”能力，我们计划将系统从“网页搜索工具”升级为“网页操作工具 (Web Operator)”。

基于系统的架构（前后端分离，后端拥有完整的系统权限），我们采取**渐进式分层架构**，重点实现“API 优先”和“AI 网页操作”，在稳定性、成本和通用性之间取得最佳平衡。

---

## 2. 方案第一层：API 优先策略 (API-First)

### 2.1 核心原理
对于主流学术资源，**优先调用官方提供的开放 API**。这种方式获取的是结构化数据（JSON/XML），速度最快、准确率最高、成本最低，且完全不存在反爬虫和沙箱限制问题。

### 2.2 主流学术资源 API 调研结果及准备要求

以下是我们首批可以接入的主流学术资源 API 调研情况：

| 学术资源平台 | 适用领域 | API 开放情况 | 接入准备与要求 | 推荐指数 |
| :--- | :--- | :--- | :--- | :--- |
| **Semantic Scholar** | 全学科综合、计算机、生物 | **极度开放** (REST API, Graph API) | **强烈建议申请免费 API Key**。<br>无 Key 时限制 100次/5分钟；有 Key 可达 100次/秒。提供文献元数据、引用网络、摘要等。 | ⭐⭐⭐⭐⭐ |
| **arXiv** | 物理、数学、计算机科学预印本 | **完全开放** (OAI-PMH / REST) | **无需 API Key**。<br>要求调用频率不超过 1次/3秒。返回 XML 格式，需在后端进行解析。 | ⭐⭐⭐⭐⭐ |
| **Crossref** | 全学科 (基于 DOI 的元数据) | **完全开放** (REST API) | **无需 API Key**。<br>建议在 HTTP Header 中提供邮箱地址进入 "Polite Pool"（礼貌池），可获得更稳定、更快的响应。 | ⭐⭐⭐⭐ |
| **PubMed (NCBI)** | 生物学、医学 | **极度开放** (E-utilities API) | **建议申请免费 API Key**。<br>无 Key 限制 3次/秒；有 Key 提升至 10次/秒。需在请求中附带邮箱。 | ⭐⭐⭐⭐ |
| **IEEE Xplore** | 计算机科学、电子工程 | **部分开放** (Metadata API) | **需注册开发者账号并申请免费 API Key**。<br>每天限制 200 次调用。仅提供元数据和摘要，不提供全文。 | ⭐⭐⭐ |
| **Springer Nature** | 综合类商业数据库 | **部分开放** (Metadata API) | **需注册申请 API Key**。<br>提供基础元数据检索，全文获取通常需要机构订阅凭证。 | ⭐⭐⭐ |
| **Elsevier (Scopus)** | 综合类商业数据库 | **限制开放** (Scopus API) | **需机构订阅 + API Key**。<br>个人开发者获取门槛较高，配额受限。 | ⭐⭐ |

### 2.3 第一层实施建议
1. **注册与鉴权准备**：立即注册 Semantic Scholar 和 IEEE Xplore 的开发者账号，获取 API Key，并存入后端的 `.env` 文件中。
2. **工具封装 (Tool Creation)**：在 `research_agent/tools/` 下新建针对上述高优 API 的工具函数（如 `search_semantic_scholar_tool`，`search_arxiv_tool`）。
3. **LLM 路由**：让 Orchestrator 能够根据用户的查询意图（如“查一下最新的大模型预印本”），自动选择调用 arXiv API 工具。

## 3. 第三层：真正的 AI 网页操作 (True AI Web Operation)

### 3.1 核心原理
对于**没有开放 API**、**需要复杂交互**（如登录、点击高级筛选、处理动态下拉框、SPA 单页应用）的网站，系统在**后端**启动 Playwright 无头浏览器，通过「简化 DOM + LLM 决策动作」的闭环完成检索。

### 3.2 核心技术栈
* **驱动引擎**：`Playwright` (Python `sync_api`)
* **感知**：`page.evaluate` 注入序号，收集可交互元素（`a/button/input/textarea/select` 等）的简短文本与 `data-ra-op-id`
* **决策**：单次 LLM 调用返回 JSON：`action` ∈ `click | type | scroll | wait | done | navigate`，及 `target_id`、`text`、`url`、`papers`（`done` 时结构化结果）

### 3.3 突破沙箱的架构设计
编排器仍只调用 `route_tool_call("web_search")`；**无头浏览器仅在** `web_search_executor` 在路由结果为 `web_operator` 时启动，与前端沙箱无关。

## 4. 代码级设计（与当前仓库对齐）

### 4.1 调用链（保持不变）

1. [`pipelines/deep/orchestrator.py`](../EPP-Backend-Dev/research_agent/pipelines/deep/orchestrator.py)：` _search_context()` → `route_tool_call(tool_name="web_search", args={"query", "url"})`
2. [`tools/router.py`](../EPP-Backend-Dev/research_agent/tools/router.py)：`web_search` → `execute_web_search(query, url)`
3. [`tools/web_search_executor.py`](../EPP-Backend-Dev/research_agent/tools/web_search_executor.py)：**唯一扩展点**——在「无显式 `url`」分支内完成学术路由与 Tavily / Web Operator 分流

显式传入 `url` 时：仍走 `allowed_get` 直连抓取（与升级前一致）。

### 4.2 新增 / 修改文件清单

| 路径 | 职责 |
|------|------|
| `research_agent/tools/academic_search_executor.py` | Semantic Scholar、arXiv、Crossref、IEEE Xplore 的 HTTP/客户端调用，统一产出 `WebSearchResult`（字段形态与 Tavily 分支一致：`citations[]` 含 `title/url/snippet/raw_content/confidence/source`） |
| `research_agent/tools/web_operator_executor.py` | Playwright 生命周期、`build_page_snapshot()`、`run_web_operator(goal, start_url, ...)` → `WebSearchResult` |
| `research_agent/tools/web_search_executor.py` | `_allowed_routes()`、`_llm_pick_search_route()`、`execute_web_search()` 中接入路由与分发 |
| `backend/settings.py` | `RA_SEMANTIC_SCHOLAR_API_KEY`、`RA_IEEE_XPLORE_API_KEY`、`RA_CROSSREF_POLITE_EMAIL`、`RA_WEB_OPERATOR_*` |
| `development.env` | 上述变量的示例占位（密钥勿提交真实值） |
| `requirements.txt` | `playwright` |

### 4.3 路由决策逻辑

1. **构造可用后端集合 `allowed`**  
   - 始终可选（无需 Key）：`arxiv`、`crossref`  
   - 配置 Key 后可选：`semantic_scholar`、`ieee_xplore`  
   - `tavily`：当 `RA_TAVILY_API_KEY`（或 `TAVILY_API_KEY` 回退）非空  
   - `web_operator`：当 `RA_WEB_OPERATOR_ENABLED` 为真且本机已安装 Playwright 浏览器（运行时检测）

2. **LLM 结构化路由**（`chat_completion`，`enable_thinking=False`）  
   - 输入：用户 `query` + `allowed` 列表说明  
   - 输出 JSON：`{"route": "<one of allowed>", "start_url": "<可选，web_operator 用>"}`  
   - 解析失败或 `route` 不在 `allowed`：**回退** `crossref`（无 Key 场景的默认学术元数据）；若 `tavily` 在 allowed 中且更偏通用检索，可由后续迭代改为 `tavily` 优先——当前实现以 `crossref` 为默认回退。

3. **无 Key 时的语义**（与产品约定一致）  
   - 未配置 Semantic Scholar / IEEE Key 时，LLM **不得**被提示选择这两项（它们不出现在 `allowed` 中）。  
   - 仍可通过 `arxiv` / `crossref` / `web_operator` / `tavily`（若已配 Tavily）满足检索。

4. **`web_operator` 的 `start_url`**  
   - 若 LLM 未给出有效 URL：从 `query` 中用正则提取首个 `http(s)` 链接；仍无则**改走** `tavily`（若可用），否则 `crossref`。

### 4.4 学术 API → `WebSearchResult` 映射约定

- **`source`**：`semantic_scholar` | `arxiv` | `crossref` | `ieee_xplore`  
- **`url`**：论文或记录详情页（IEEE 用 `html_url` 或 `pdf_url` 优先 HTML）  
- **`snippet`**：摘要或短描述（截断至 ~300 字）  
- **`raw_content`**：`truncate_text(..., 2200)` 内的标题+作者+年份+摘要拼接  
- **`confidence`**：字符串浮点；API 无评分时统一 `"0.75"` 或按排序递减  
- **审计**：`make_audit("web_search", ...)`，`metadata` 中附带 `provider=...`

### 4.5 Web Operator 生命周期

1. `sync_playwright().start()` → `browser.launch(headless=True)` → `new_page()`  
2. `page.goto(start_url)`（超时见 `RA_WEB_OPERATOR_GOTO_TIMEOUT`）  
3. 循环至多 `RA_WEB_OPERATOR_MAX_STEPS` 次：  
   - `build_page_snapshot(page)` → 文本块给 LLM  
   - 执行返回的 `click/type/scroll/wait/navigate`  
   - `done`：将 `papers` 转为 `citations` 并返回  
4. `browser.close()` + `playwright.stop()`（使用 `try/finally` 保证释放，仅此一处需要成对清理）

### 4.6 运维提示

- 首次部署需执行：`playwright install chromium`（或等价仅安装 chromium）  
- 出站域名仍受 [`site_access_control`](../EPP-Backend-Dev/research_agent/site_access_control.py) 约束；Web Operator 访问的 `start_url` 主机须在允许列表内  

---

## 5. 开发原则（本模块）

- 不把简单逻辑复杂化；避免「永远走不到」的 `except`  
- 不做旧版本兼容分支  
- 优先在数据流的真实边界改（本设计中边界为 `execute_web_search` + 新增 executor），不扩散到 `orchestrator` 除非契约变更
