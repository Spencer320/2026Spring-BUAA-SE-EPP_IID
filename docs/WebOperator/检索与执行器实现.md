# 检索与执行器实现 (Search & Executor Implementation)

## 1. 概述
在 Research Agent 中，为了满足不同场景下的信息获取需求，系统设计了多层次的检索与执行器架构。主要包括通用网页搜索、学术文献专用检索，以及基于无头浏览器的网页操作代理（Web Operator）。这些执行器统一封装，通过路由机制智能选择最合适的工具。

## 2. 核心执行器介绍

### 2.1 学术搜索执行器 (`academic_search_executor.py`)
专门针对学术文献数据库的 API 检索实现，确保获取高质量的学术资源。
- **支持的来源**：
  - **Semantic Scholar**：通过 Graph API 获取论文元数据（标题、作者、摘要、URL）。
  - **arXiv**：使用 `arxiv` Python 库检索预印本。
  - **Crossref**：通过 Crossref REST API 检索广泛的学术出版物（利用 DOI 和链接）。
  - **IEEE Xplore**：通过官方 API 检索 IEEE 文献。
- **实现特点**：
  - 统一返回 `WebSearchResult` 结构，包含规范化的引用信息（`citations`）。
  - 自动提取和截断摘要（`truncate_text`），计算置信度（`confidence`），并提取域名。
  - 具备完善的错误处理和审计日志（`make_audit`）。

### 2.2 网页操作代理 (`web_operator_executor.py`)
一个基于 Playwright 和 LLM 结合的自主网页操作雏形，用于处理需要交互（如点击、翻页、搜索框输入）的复杂网页。
- **工作原理**：
  1. 使用 Playwright 启动无头 Chromium 浏览器并访问目标 URL。
  2. 注入 JavaScript 脚本（`_SNAPSHOT_JS`）提取页面当前可见的交互元素（a, button, input 等），并分配唯一 ID。
  3. 将“页面可操作元素列表”和“用户目标”发送给 LLM（`chat_completion`）。
  4. LLM 决定下一步动作（`action`），如 `click`、`type`、`scroll`、`navigate`、`wait` 或 `done`。
  5. 循环执行，直到 LLM 输出 `done` 并提取出所需的论文/信息条目，或达到最大步数限制（`RA_WEB_OPERATOR_MAX_STEPS`）。
- **应用场景**：当目标网站没有公开 API，或者用户明确要求在特定网站（如知网）内进行深度搜索时。

### 2.3 综合网页搜索与路由 (`web_search_executor.py`)
作为搜索的统一入口，负责通用搜索（Tavily）以及各专业执行器的智能路由。
- **智能路由 (`_llm_pick_search_route`)**：
  - 使用 LLM 分析用户的 Query，从允许的路由列表（如 arxiv, crossref, semantic_scholar, tavily, web_operator）中选择最合适的一个。
  - 启发式规则：如果用户 Query 包含特定 URL 或特定站点关键词（如“知网”、“IEEE Xplore”），或者包含“网页操作”、“打开网站”等提示词，系统会优先路由到 `web_operator`。
- **Tavily 搜索**：
  - 支持中英双语扩展查询（`_bilingual_queries`），提高检索覆盖率。
  - 执行多范围（Whitelist 优先 + Global 兜底）的并发或顺序检索。
- **结果后处理**：
  - **去重与排序 (`_dedupe_and_rank_citations`)**：根据 URL 和标题去重，结合置信度评分和学术域名白名单（如 arxiv.org, nature.com）进行加权排序。
  - **站点策略过滤 (`_filter_citations_by_site_policy`)**：确保返回的链接符合系统的出站访问控制策略。
  - **容错与降级 (`_try_search_routes`)**：如果首选路由失败，系统会按照预定优先级（如 primary -> crossref -> arxiv -> tavily）依次尝试其他路由，保证搜索的鲁棒性。

## 3. 使用方法与触发机制

### 3.1 学术 API 配置说明
学术 API 的密钥在后端的环境变量文件（如 `development.env` 或系统环境变量）中进行配置。
- **配置位置**：
  - `RA_SEMANTIC_SCHOLAR_API_KEY`: Semantic Scholar API 密钥。
  - `RA_IEEE_XPLORE_API_KEY`: IEEE Xplore API 密钥。
  - `RA_TAVILY_API_KEY`: Tavily 通用搜索 API 密钥（必须配置，否则无法进行基础的网络搜索）。
  - `RA_CROSSREF_POLITE_EMAIL`: Crossref 的 Polite Pool 邮箱（建议配置以获得更稳定的 API 访问）。
- **未配置的影响**：
  - 如果未配置某个特定的学术 API（如 IEEE Xplore），系统在路由时会自动将其从可用列表中剔除。
  - 当用户查询需要该 API 时，系统会自动降级（Fallback）到其他可用的学术 API（如 arXiv, Crossref）或通用搜索引擎（Tavily）。
  - arXiv 和 Crossref 默认不需要 API Key 即可使用，因此即使不配置专属 Key，系统仍具备基础的学术检索能力。

### 3.2 Web Operator (Playwright) 现状说明
Web Operator 目前是一个**未完全确认可行性的实验性雏形**。
- **定位**：旨在解决传统 API 无法覆盖的封闭网站（如需要复杂交互、动态渲染或特定权限的站点，例如中国知网 CNKI）。
- **现状**：
  - 依赖于 Playwright 库启动无头浏览器，并通过注入 JS 脚本提取页面元素。
  - 强依赖 LLM 的推理能力来决定下一步操作（点击、输入、滚动等）。
  - **局限性**：由于网页结构复杂多变、反爬虫机制（如验证码）、以及 LLM 推理的不稳定性，Web Operator 在实际运行中可能会遇到死循环、元素定位失败或超时等问题。目前建议仅作为备用方案或在特定受控站点下进行测试。

### 3.3 执行器触发场景与用户输入示例
系统通过 LLM 路由（`_llm_pick_search_route`）和启发式规则（关键词匹配）来决定触发哪个执行器。

- **触发通用学术搜索 (arXiv, Crossref, Semantic Scholar)**
  - **场景**：用户进行常规的学术概念、技术原理或文献综述查询，没有指定特定网站。
  - **用户输入示例**：
    - *"帮我查找关于大语言模型中 RAG 技术的最新研究论文。"*
    - *"什么是 Transformer 架构？请列出相关的经典文献。"*

- **触发特定学术库 (IEEE Xplore)**
  - **场景**：用户明确要求在特定学术库中查找，且系统已配置该库的 API。
  - **用户输入示例**：
    - *"在 IEEE Xplore 上搜索关于 6G 通信物理层的论文。"*

- **触发 Web Operator (网页操作代理)**
  - **场景**：用户明确提供了一个具体的 URL 要求深入浏览，或者提到了需要复杂交互的封闭站点（如知网），亦或使用了强烈的“网页操作”暗示词。
  - **用户输入示例**：
    - *"打开知网，搜索‘深度学习在医疗领域的应用’，并提取前5篇论文的摘要。"*
    - *"访问这个网址 https://example.com/papers，点击搜索框输入‘量子计算’，然后把结果整理给我。"*
    - *"请用无头浏览器帮我进入 Semantic Scholar 的主页进行高级搜索。"*

- **触发通用网页搜索 (Tavily)**
  - **场景**：用户的查询偏向于时事、新闻、非学术类的通用知识，或者学术 API 均未找到相关结果时的兜底方案。
  - **用户输入示例**：
    - *"2026年最新发布的 AI 模型有哪些？"*
    - *"帮我查一下最近关于科技公司裁员的新闻报道。"*
