# 深度研究编排器

本文描述 **科研助手模块**（`research_agent`）中的「独立深度研究」编排器，与 `basic_orchestrator`、工作区管线分离。  
与根路径 **`/api/deep-research/`**（`business` 包）是另一套产品，不要混用。

---

## 1. 核心实体与入口

| 项 | 说明 |
|----|------|
| 持久化模型 | `AgentTask`（`related_name="deep_research_tasks"`） |
| 执行函数 | `orchestrator.execute_deep_research_pipeline` |
| 阶段常量 | `PIPELINE_PHASES = ("plan", "decide", "search", "read", "reflect", "write")` |
| 线程入口 | `start_deep_research_thread` → 非 sqlite 时在后台线程执行流水线 |

---

## 2. 如何触发

### 2.1 HTTP 入口

| 入口 | 说明 |
|------|------|
| **HTTP** | `POST /api/research-agent/tasks/deep-research/`（`research_agent/urls.py` → `views.create_deep_research_task`） |
| **批准后继续** | `execute_after_approve`：若 task id 对应 `AgentTask`，则再次进入 `execute_deep_research_pipeline`（例如 `pending_action` 批准后） |
| **修订后继续** | `start_after_revise_thread` → `execute_after_revise` → 仅深度任务会续跑流水线 |

### 2.2 并发限制

同一会话若已有 `pending` / `running` / `pending_action` 的 **Basic 或深度** 任务，再发普通任务或深度研究会 **409**（`views._active_task` 同时检查 `basic_runs` 与 `deep_research_tasks`）。

### 2.3 前端科研助手页

`ResearchAgentSession.vue` 当前要求：**有提示词 + `session_id`**，调用 `createDeepResearchTask`。`selected_papers` 为可选字段，若提供则须是展示区已有条目。

---

## 3. 输入是什么

### 3.1 请求体（`create_deep_research_task` + `_validate_task_options`）

| 字段 | 说明 |
|------|------|
| `content` 或 `query` | 用户研究提示词，**必填**（经 `_normalize_query_field`） |
| `session_id` | 有 **`selected_papers` 且非空** 时 **必填**，且会话须属于当前用户 |
| `selected_papers` | 可选；若非空，每项须是会话 **论文展示区** 已有条目的 UUID（`_validate_selected_papers_from_shelf`），最多 50 条；入口规范化为带 `title`、`primary_url`、`workspace_rel_path`、`context_tier` 等的字典列表 |
| 其它 runtime 选项 | 如 `risk_confirmation_strategy`、`max_reflect_rounds`（1–5）、可选 `local_command`、`local_file_action` 等，写入 `result_payload.runtime_config` |

### 3.2 任务创建时落库（`_start_deep_research_task`）

- 写两条 `ResearchMessage`：一条 **user**（纯 `content`，不含结构化 papers）、一条助手「已收到深度研究请求…」。
- 创建 `AgentTask`，`result_payload = { "runtime_config": { ..., "deep_research_pipeline": true, "selected_papers": [...] } }`。

### 3.3 编排器内实际喂给模型的「用户问题」字符串

1. `query = _latest_user_query(task)`：**当前会话里按时间倒序最新一条 `role=user` 的 `ResearchMessage.content`**。
2. `query = _deep_research_augment_user_query(task, query)`：若 `runtime_config.selected_papers` 非空，把其 JSON（截断约 8k）拼在 `query` 后，整段再截断约 **50k** 字符，作为各阶段 LLM 里 `USER_PROMPT_*` 中的 `query` 占位符来源。

---

## 4. 执行链路

### 4.1 宏观顺序（`execute_deep_research_pipeline`）

1. **plan**：一次 LLM → 多方案规划 JSON；失败则硬兜底；`_append_step(..., "plan", ...)`。
2. **decide**：一次 LLM → 选方案并拆 `subtasks`；校验/兜底；`_append_step(..., "decide", ...)`。
3. **对每个 subtask** 循环（内层还有 **reflect 多轮**）：
   - **`_search_context(subtask_goal)`**：先做 **工具侧检索**（如 `web_search` 路由），得到 `citations` 等；若 **fatal** 则任务失败并 return。
   - **search（LLM）**：把 `query`、子任务标题、检索结果 JSON、上一轮 reflect 的 `feedback` 等塞进 `USER_PROMPT_SEARCH`；产出结构化 `search_payload`（含 `info_groups`）；多轮校验/修复；`_append_step(..., "search", ...)`。
   - **可选 `local_command` / `local_file_action`**（`_maybe_run_local_command` / `_maybe_run_local_file_action`）：可能把任务置为 **`pending_action`** 并写入 `intervention`，流水线 **中途 return**，等用户批准后 `execute_after_approve` 再续跑。
   - **read（LLM）**：消费 `info_groups` + 引用渲染；`_append_step(..., "read", ...)`。
   - **reflect（LLM）**：若 `needs_optimization == "yes"` 且未达 `max_reflect_rounds`，带着 suggestions 再搜/再读一轮；否则把本 subtask 的 `accepted_reader_summary` 等并入 **`final_subtask_summaries`**。
4. **write**：一次 LLM，输入含 `query`、`decision_payload`、`final_subtask_summaries`、`all_reflector_conclusions`；产出 `write_payload`，再经 `_markdown_from_write_json` 得到 **`report_body`**；`_append_step(..., "write", ...)`。
5. 成功：`task.status = "completed"`，写入完整 `result_payload`；并新建一条 **assistant** `ResearchMessage`，内容为 **`[[RA_REPORT]]\n` + `report_body`**。

### 4.2 与其它编排器的关系（边界）

- **Basic / Workspace**：`basic_orchestrator` 与 `execute_deep_research_pipeline` **互不调用**。
- 工作区多轮在 `workspace_pipeline`，**不在**本流水线。
- 深度研究若需本地文件能力，走的是 orchestrator 里「本地命令 / 本地文件」工具链（可能产生 `pending_action`），与 `WorkspaceAgentRun` 不是同一条线。

## 5. 输出是什么

### 5.1 谁产出

| 来源 | 产出 |
|------|------|
| **LLM**（`chat_completion`） | 各阶段结构化中间结果（plan / decide / search / read / reflect / write JSON） |
| **工具**（`_search_context` → `route_tool_call("web_search", ...)`） | 检索事实与 **citations**，在子任务循环中累积到 **`all_citations`** |
| **本地命令/文件** | 工具路由结果，可能触发 **`pending_action`**，中断或续跑流水线 |
| **编排器** | 拼装 prompt、校验 JSON、失败兜底、写 **`AgentTask.steps`**、写 **`result_payload`**、写 **`ResearchMessage`**、写 **`AgentBehaviorAuditLog`**（经 `_append_step` → `_append_behavior_log`） |

### 5.2 逐步对外可见产物

| 位置 | 内容 |
|------|------|
| **`task.steps`** | 每步一条：`seq`、`phase`（plan/decide/search/read/reflect/write）、`title`、`detail`、`ts`；用于轮询展示进度与摘要 |
| **`task.result_payload`（完成后）** | `format: "markdown"`，`body`：最终报告 Markdown；`citations`；`attachments`（当前固定 `[]`）；以及 `pipeline`、`reflect_rounds`、`planner_alternatives`、`decider_decision`、`subtask_summaries`、`all_reflector_conclusions`、`applied_suggestions`、`runtime_config` 等溯源/调试结构 |
| **`ResearchMessage`（assistant）** | `[[RA_REPORT]]` 前缀 + 与 `result_payload.body` 同源的报告正文，便于会话时间线展示 |

### 5.3 各阶段与最终报告的关系

| 阶段 | 作用 |
|------|------|
| **plan / decide** | 决定子任务拆分与目标 **`subtask_goal`**，约束后面每轮的检索与阅读范围 |
| **search（工具 + LLM）** | 工具给 **citations**；LLM 整理为 **info_groups**，供 read 消化 |
| **read** | 把单轮检索材料压成 **`read_payload`**，进入 reflect |
| **reflect** | 决定是否再搜，并产出 **`accepted_reader_summary`** 等，汇总进 **`final_subtask_summaries`** 与 **`all_reflector_conclusions`** |
| **write** | 把 **决策 JSON + 各子任务最终摘要 + 全部 reflect 结论** 合成 **`write_payload`**，渲染为 **最终 Markdown `body`**；**`all_citations`** 作为并列字段挂在 `result_payload` 里 |
