# 深度研究链路优化分析

本文从**输入/输出流**与**提示词设计意图**两个维度审视六阶段流水线（plan → decide → search → read → reflect → write），判断各阶段存在的必要性，并给出具体的优化建议。每条建议均附有「为什么优化」「合理性论证」「预期效果」。

---

## 0. 六阶段逐一审视：哪些阶段不合理？

### 0.1 plan + decide：双阶段串行是伪需求

**现状**：

- **plan** 阶段（`USER_PROMPT_PLAN`）：LLM 产出 2-4 个备选方案（`alternatives`），含 `plan_id/title/steps/rationale`。
- **decide** 阶段（`USER_PROMPT_DECIDE`）：LLM 从 `alternatives` 中选择一个最优方案，输出 `selected_plan_id/decision_reason/complexity/subtasks`。

**问题分析**：

1. **备选方案对用户不可见，是纯中间产物**。plan 产出的 2-4 个方案中，只有被 decide 选中的那一个会进入后续流程，其余方案全部丢弃。这相当于让 LLM "头脑风暴"但只用一句话。

2. **decide 完全重复了 plan 的推理**。plan 阶段 LLM 已经对用户 query 做了深层理解并生成了结构化方案；decide 阶段再次输入相同 `user_query` + `alternatives_json`，要求 LLM 做选择——这个"选择"本质上等价于"在生成方案时就确定最优的那个"，中间没有新信息注入。

3. **plan 的 `rationale` 和 decide 的 `decision_reason` 语义重叠**。前者解释每个方案的合理性，后者解释为什么选某个方案。二者内容高度重复。

4. **复杂度判定（simple vs complex）在 plan 阶段已经隐式确定**：备选方案的 `steps` 数量本身就反映了复杂度，decide 显式要求 `complexity` 字段是冗余判断。

**结论**：plan + decide 应合并为单次 LLM 调用。plan 的「多方案生成」是过度设计——学术研究场景下，用户需要一个确定的执行计划，而非让 LLM 扮演委员会投票。

### 0.2 search（LLM）：定位偏离——是"整理"而非"搜索"

**现状**：

- 工具侧：`_search_context(subtask_goal)` → `route_tool_call("web_search", ...)` 执行真正的网络检索，返回 `citations` 和结构化搜索结果。
- LLM 侧（`USER_PROMPT_SEARCH`）：`role=searcher`，提示词明确要求「对传入的 raw_search_results 进行清洗、去重和分组，只输出整理后的信息分组，不做总结性结论」。

**问题分析**：

1. **名称与实质不符**。该阶段叫作"search"，但 LLM 不执行任何搜索——它只是**格式化整理**工具返回的原始结果。名称误导，让链路看起来比实际更复杂。

2. **"禁止做总结性结论"与后续 read 阶段形成冗余切分**。searcher 被要求只整理不总结 → 紧接着 reader 被告知「仅基于输入做归纳阅读」。两个阶段处理的是**同一批数据**（`info_groups`），区别仅仅是「分组」与「分析」的二分——而 LLM 实际上在一次调用中就能同时完成两者。

3. **`info_groups` 的格式化工作大部分是机械性的**。工具返回的 JSON 本身已有结构（标题、URL、摘要），searcher 按 `group_title/relevance/raw_findings/sources` 重新组织，这项工作完全可以合并到 read 阶段或由代码预处理。

**结论**：searcher 的 LLM 调用是**低信息增益的格式化步骤**。其输入（`raw_search_results`）和输出（`info_groups`）之间的语义差异极小。应与 read 合并。

### 0.3 read：依赖 searcher 的输出，但二阶段可合并

**现状**（`USER_PROMPT_READ`）：

- `role=reader`，输入为 `info_groups` + `citation_context`。
- 产出 `analysis/key_points/limitations/references`，要求使用 `[1]/[2]` 角标引用真实来源。

**问题分析**：

1. **reader 是真正的「理解与归纳」阶段**，这在逻辑上合理。但其输入完全由 searcher 准备，而 searcher 的格式化工作是低价值的。

2. **reader 的 `key_points` + `limitations` 输出与 searcher 的 `info_groups` 之间的信息梯度，远小于从 raw_search_results 到最终分析之间的总梯度**。换言之，searcher 是一个不必要的中间层。

**结论**：read 阶段本身合理，但应直接接收工具返回的原始检索结果，跳过 searcher LLM。

### 0.4 reflect：输出大量重复内容，价值集中在单一布尔判断

**现状**（`USER_PROMPT_REFLECT`）：

- 输入：`reader_summary`（read 阶段的完整输出 JSON）。
- 输出：`needs_optimization( yes|no )`、`reason`、`actionable_suggestions`、`accepted_reader_summary`（完整嵌套的 reader 输出结构）。

**问题分析**：

1. **`accepted_reader_summary` 是 reader 输出的几乎原样复制**。reflect prompt 明确要求 `accepted_reader_summary` 必须包含 `analysis/key_points/limitations/references`，与 reader 输出结构完全相同。当 `needs_optimization=no` 时，LLM 被要求把 reader 的 JSON 原封不动嵌套在 reflector 的 JSON 里——这是纯粹的 token 浪费。

2. **reflect 真正有价值的信息只有三个字段**：`needs_optimization`（1 个 token）、`reason`（几句话）、`actionable_suggestions`（几条建议）。但为获得这些，需要把 reader 的完整输出重新喂给 LLM 并要求其重复输出。

3. **reflect 的 prompt 中 `当前轮次：{reflect_round}/{max_rounds}` 信息量极低**。LLM 不需要知道「当前是第几轮」也能判断是否需要优化——它的判断依据是 reader 输出的充分性，而非轮次号。

4. **轮次判断偏差**：reflector 知道自己在第几轮，可能产生策略性行为——早期轮次倾向说"需要优化"以显得尽责，末期轮次倾向说"不需要"以结束循环。这并不反映真实的信息充分性。

**结论**：reflect 的核心功能（是否需要继续搜索）可以用更轻量的方式实现。若保留 reflect 阶段，应去掉 `accepted_reader_summary` 的冗余输出。

### 0.5 write：六阶段中唯一不可替代的阶段

write 阶段负责将各子任务最终摘要合成为用户可读的 Markdown 报告。这个阶段的设计是合理的：
- 输入是所有子任务的结构化结论，输出是最终用户可见产物。
- `sections + traceability + references` 的 schema 设计良好，保证报告可溯源。

但 write 阶段仍存在上下文膨胀问题（见优化建议 3）。

## 1. 优化建议一：合并 plan + decide 为单阶段「规划决策」

### 为什么优化这一条？

当前 plan 和 decide 是两个串行的 LLM 调用，但其信息流本质是：

```
plan: query → alternatives (2-4 个方案)
decide: query + alternatives → selected_plan + subtasks
```

两个阶段共享相同的 `user_query`，decide 的输入完全由 plan 的输出决定，中间无外部信息注入。这在系统工程上是典型的**可合并串行步骤**——前一步的输出仅作为后一步的输入，没有人类干预或外部数据参与决策。

### 优化的合理性

1. **LLM 完全有能力在单次调用中完成「方案构思 + 选择 + 子任务拆分」**。当前 prompt 已经要求 plan 生成 `rationale`（解释每个方案的优劣）和 decide 生成 `decision_reason`（解释选择依据）——这本质上是同一段推理的自然延伸，人为切成两段只会迫使 LLM 在 decide 阶段"回忆" plan 阶段的推理。

2. **学术研究场景不需要多方案对比**。用户提交深度研究时，期望的是一个确定的执行计划，而非 2-4 个待选的抽象方案。用户不参与方案选择（plan/decide 之间无 `pending_action` 中断），多方案生成纯属内耗。

3. **plan 的备选方案可以降级为可选溯源字段**，而非主链路必经步骤。保留在 `result_payload.planner_alternatives` 中供调试即可。

4. **实测验证**：当前代码中 `_coerce_decider_complexity` 函数的存在本身就说明 decide 输出不稳定（模型可能输出中文/英文/布尔值混合的 `complexity`），需要额外兜底。合并后只需一个 `complexity` 字段，减少兜底复杂度。

### 会达到什么效果？

- **延迟降低**：消除一个完整的 LLM 往返（~15-50s，取决于模型和网络），流水线总延迟降低约 35-40%。
- **Token 消耗减少**：不再需要把 `alternatives_json` 再次输入给 decide，也不再需要 decide 输出重复的 `decision_reason`。单次深度研究可节省约 2000-5000 token。
- **代码简化**：不再需要 `_coerce_decider_payload`、`_pick_decider_json`、两段独立的 `_call_llm_with_fallback` 逻辑。
- **不影响输出质量**：合并后的 prompt 可以要求 LLM 在内部推理中对比方案（但不输出），仅输出最终选定方案。思维链内化到单次调用中，不损失推理深度。

## 2. 优化建议二：合并 search（LLM）+ read 为单阶段「检索整理与深度阅读」

### 为什么优化这一条？

当前 search（LLM）的 prompt 明确限定角色为：

> "你是信息整理角色，请对传入的 raw_search_results 进行清洗、去重和分组，只输出整理后的信息分组，**不做总结性结论**。"

紧接着 read 阶段：

> "仅基于输入做归纳阅读，不新增外部来源。"

两个阶段处理的是同一批原始数据，只是被人工切分为「格式化分组」和「归纳分析」两步。这导致：

1. **search（LLM）产生低信息增益**：输入是工具返回的 JSON（已有标题/URL/摘要），输出是按 `group_title/relevance/raw_findings/sources` 重新组织的 JSON。语义内容几乎不变，仅改变 JSON 结构——这项工作完全可以用代码预处理（按来源域名分组、按关键词匹配度排序等），无需 LLM。

2. **search 的 `raw_findings` 和 read 的 `key_points` 语义边界模糊**：search 的 `raw_findings` 是「从原始结果提取的关键发现」，read 的 `key_points` 是「关键要点」。区别仅在于前者不允许跨分组综合，后者允许。但 LLM 在实际执行中往往已经在 raw_findings 里做了轻度总结（因为「提取」本质上就是归纳），导致 read 阶段面对的是已经被轻度总结过的内容，再做一次深层总结——信息衰减。

3. **search（LLM）的禁止性约束反而增加了 read 的负担**："禁止做总结性结论" 意味着 searcher 即便发现了信息之间的关联，也不能表达出来。这些洞察被埋没在 search 阶段，read 阶段需要从分散的 info_groups 中重新发现——这是一次浪费的认知重置。

### 优化的合理性

1. **LLM 天然适合同时完成「整理 + 分析」**。阅读理解和信息组织是同一认知过程的不同层次，强制分离反而降低效率。合并后的 prompt 可以要求 LLM 直接输出 `analysis/key_points/limitations/references`，不需要中间的 `info_groups` 层。

2. **工具返回的搜索结果已经结构化**（`route_tool_call("web_search")` 返回带 URL/snippet 的数据），可以通过代码预处理完成基本的去重、按域名分组、过滤死链，不需要 LLM 介入。

3. **减少一次 JSON 解析/校验/兜底的工程开销**。当前代码中 searcher 输出需要通过 `_ensure_searcher_minimal_groups`、`_coerce_searcher_payload`、`pick_searcher_json_payload` 等多层兜底。合并后只需一个 reader 的兜底逻辑。

4. **提升引用质量**：合并后 LLM 直接从原始搜索结果中提取引用，减少 info_groups 中间层可能引入的引用丢失或变形。当前链路中 citation 从工具侧 → searcher → reader 传递，每层都可能丢失部分信息。

### 会达到什么效果？

- **每个子任务减少一次 LLM 调用**：从「search LLM + read LLM」变为单次「analyze LLM」。对于 3 子任务 × 3 reflect 轮的典型场景，节省 9 次 LLM 调用。
- **分析深度提升**：LLM 在一次调用中同时看到原始检索结果和完成分析任务，可以在组织信息的同时做深层推理，而非先做浅层分组再做深层分析。
- **Token 消耗减少**：不再需要把 info_groups JSON 序列化后再输入给 read。工具返回的原始结果语义密度更高（没有被 search 阶段截断/改写）。
- **引用完整性更好**：从工具到分析的路径更短，引用信息丢失概率降低。

## 3. 优化建议三：精简 reflect 阶段，去除 `accepted_reader_summary` 冗余输出

### 为什么优化这一条？

当前 reflect 阶段的输出 schema 要求：

```json
{
  "needs_optimization": "yes|no",
  "reason": "...",
  "actionable_suggestions": ["..."],
  "accepted_reader_summary": {
    "analysis": "...",
    "key_points": ["..."],
    "limitations": ["..."],
    "references": [{"id": 1, "title": "...", "url": "..."}]
  }
}
```

其中 `accepted_reader_summary` 的结构与 reader 输出**完全相同**。当 `needs_optimization=no` 时，LLM 被要求「原样保留 reader_summary」——这意味着一大段 JSON 被**原封不动地从输入复制到输出**。这是纯粹的 token 浪费，无任何信息增量。

当 `needs_optimization=yes` 时，`accepted_reader_summary` 可能包含 reflector 对 reader 输出的微调——但这种微调在下轮 search→read 循环中会被**覆盖**（新 reader 输出会替代旧 summary），并不会被保留。

### 优化的合理性

1. **reflect 的核心价值是「是否继续」的决策**，而非「复制 reader 输出」。从信息论角度，reflect 输入（reader_summary JSON）和输出（accepted_reader_summary JSON）之间的互信息接近 100%——几乎所有信息都是重复的。

2. **当下轮 `needs_optimization=yes` 时，`accepted_reader_summary` 的使用方式是**：不直接使用，而是把 `actionable_suggestions` 传给 searcher 的 `previous_reflector_feedback`。而 reader 的原始输出已经被记录在 `task.steps` 中，不需要 reflector 再复制一遍。

3. **最终报告使用的摘要来源是 `final_subtask_summaries`**，这个列表由每轮 reflect 的 `accepted_reader_summary` 累积。但如果 reflector 不再输出这个字段，可以直接使用最后一轮 reader 的输出作为该子任务的最终摘要——因为最后一轮必定是 `needs_optimization=no` 的 reader 输出，更完整。

4. **`reason` 和 `actionable_suggestions` 质量存疑**：当前 prompt 对这两项的约束极弱（"评估是否需要继续优化"），reflector 经常输出泛泛的建议（"需要更多数据""搜索更具体的关键词"），这些建议对下一轮 search 的实际指导价值有限。

### 会达到什么效果？

- **每个 reflect 调用节省 ~60-80% 输出 token**：去除 `accepted_reader_summary` 后，reflector 只需输出约 200-500 token（`needs_optimization + reason + suggestions`），而非 2000-5000 token。
- **简化 `final_subtask_summaries` 的构建逻辑**：直接使用最后一轮 reader 输出，而非从 reflector 中提取。
- **代码简化**：去除 `pick_reflector_json_payload` 中对 `accepted_reader_summary` 的强制校验，减少 `_struct_json_diag` 噪音。
- **不损失任何功能**：`needs_optimization + actionable_suggestions` 已足够驱动 reflect 循环；reader 输出已保存在 `task.steps` 中供审计。

## 4. 优化建议四（补充）：search 阶段尝试「内容去重 + 可信度预标注」

### 为什么优化这一条？

当前工具侧检索（`_search_context` → `route_tool_call("web_search")`）返回原始结果后，直接将其序列化为 JSON 字符串传给 search（LLM）阶段。这中间缺少两个关键预处理：

1. **内容去重**：不同搜索引擎可能返回同一篇论文的不同镜像 URL，或被不同网站转载的同一内容。LLM 需要自行判断去重，但其基于文本相似度的判断不如 URL 规范化 + 文本哈希可靠。
2. **来源可信度评估**：工具返回的结果混杂了 arXiv 预印本、个人博客、新闻报道、知乎回答等。LLM 在做 info_groups 时，所有来源被平等对待。这可能导致后续 read 阶段将低可信度来源与权威来源等权重处理。

### 优化的合理性

1. **代码侧去重比 LLM 更精确且零 token 消耗**：可以基于 URL 规范化（去 `www.`、去 query params、统一协议）和内容 simhash 做去重，在检索结果进入 LLM 之前完成。

2. **可信度预标注可以引导 LLM 的注意力分配**：在输入 `raw_search_results` 时，为每条结果增加 `credibility: "high"|"medium"|"low"|"unknown"` 标签（如 arXiv/PubMed/IEEE 标记为 high，个人博客标记为 low）。LLM 在 organize 和 analyze 时会自然倾向于高可信度来源，提升最终报告质量。

3. **学术场景的特殊性**：深度研究的目标用户是科研人员，对信息来源的权威性高度敏感。当前系统把知乎回答和 Nature 论文混在一起交给 LLM 处理，不符合科研场景的严肃性要求。

### 会达到什么效果？

- **信息质量提升**：LLM 获得的检索结果更干净、去重，减少在重复信息上的 token 浪费。
- **报告可信度提升**：write 阶段的 references 将更集中于权威来源，降低低质量引用比例。
- **工程成本极低**：去重和可信度预标注是纯代码实现（正则 + 域名白名单），不消耗 LLM token。

---

## 5. 总结：从六阶段到四阶段的优化方案

综合以上分析，建议将当前六阶段流水线重构为四阶段：

| 当前阶段 | 优化后 | 变更说明 |
|----------|--------|----------|
| plan + decide | **plan** | 合并为单次「规划决策」LLM 调用 |
| search（工具） | **search（工具）** | 保留，增加去重 + 可信度预标注 |
| search（LLM）+ read | **read** | LLM 直接消费工具检索结果，同时完成整理与分析 |
| reflect | **reflect** | 精简输出，仅保留 `needs_optimization + reason + suggestions` |
| write | **write** | 保留，增加输入压缩（见前文分析） |

**预期总收益**：

- 流水线 LLM 调用次数从 **6 + N×(3 + R×3)** 降至 **4 + N×(2 + R×2)**（N=子任务数，R=reflect 轮数），典型场景（3 子任务 × 2 reflect 轮）从 27 次降至 19 次。
- 总 token 消耗降低约 40-50%。
- 代码复杂度降低：消除 plan/decide 衔接逻辑、searcher 兜底逻辑、reflector 冗余校验。
- 报告质量不降反升：减少中间格式化层数，信息传递路径更短，引用更完整。
