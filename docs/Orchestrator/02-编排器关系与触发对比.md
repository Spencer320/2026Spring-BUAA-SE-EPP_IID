# 编排器关系与触发对比

本文说明 **科研助手模块**（`research_agent`）内三套编排器之间的区别、关系与触发差异。  
范围：`AgentTask`（深度研究）、`BasicOrchestratorRun`（会话主编排）、`WorkspaceAgentRun`（工作区子运行）。

---

## 1. 三套编排器一览

| 模型 | 含义 | 执行入口 | 用户能否单独 HTTP 启动 |
|------|------|----------|------------------------|
| `AgentTask` | **独立深度研究**六阶段流水线 | `orchestrator.execute_deep_research_pipeline` | **能**：`POST /api/research-agent/tasks/deep-research/` |
| `BasicOrchestratorRun` | **会话主编排**（日常对话 / 检索 / 委托工作区） | `basic_orchestrator.execute_basic_pipeline` | **能**：`POST /tasks/`、`POST /sessions/messages/`、`POST /sessions/<id>/messages/`、`POST /tasks/<id>/follow-up/` 等 |
| `WorkspaceAgentRun` | **工作区子运行**（多轮工具循环） | `agent_orchestrator.run_workspace_delegate` → `workspace_pipeline.execute_workspace_pipeline` | **不能**：仅由 Basic 的 `agent` 子步 **同步** 委托拉起 |

---

## 2. 关系（谁调用谁）

```
用户 HTTP
    │
    ├─ POST /tasks/deep-research/ ──────────────► AgentTask ──► execute_deep_research_pipeline
    │                                              （与 Basic / Workspace 无调用关系）
    │
    └─ POST /tasks/、/sessions/.../messages/ 等 ─► BasicOrchestratorRun ──► execute_basic_pipeline
                                                      │
                                                      │ Smart Planner 若排了 type=agent 子步
                                                      ▼
                                              run_workspace_delegate（同步）
                                                      │
                                                      ▼
                                              WorkspaceAgentRun ──► execute_workspace_pipeline
```

要点：

- **深度研究** 与 **Basic** 互不调用；两条链平行，共用同一会话与论文展示区，但运行实体不同。
- **Workspace** 不是第三条用户入口，而是 Basic 内部的子运行；`parent_basic_run` 指向父 Basic，便于审计与 `workspace_fs_generation` 同步。
- 迁移 `0007_split_orchestration_runs` 后：**`AgentTask` 不再承载 basic / workspace 状态**。

---

## 3. 触发区别

### 3.1 深度研究（`AgentTask`）

| 用户操作 | HTTP |
|----------|------|
| 明确走深度研究；可选用当前会话论文展示区条目 | `POST /api/research-agent/tasks/deep-research/` |

- `selected_papers` 非空时必须带合法 `session_id`。
- 后台：`start_deep_research_thread` → `execute_deep_research_pipeline`。
- `runtime_config` 含 `deep_research_pipeline: true`；**不会**走 Smart Planner。

### 3.2 日常科研助手（`BasicOrchestratorRun`）

| 用户操作 | HTTP |
|----------|------|
| 直接开任务（可顺带新建会话） | `POST /tasks/` |
| 新建会话并带第一句话 | `POST /sessions/messages/` |
| 在已有会话里发消息 | `POST /sessions/<session_id>/messages/` |
| 上一轮 Basic 已 **completed** 后接着问 | `POST /tasks/<父任务 id>/follow-up/` |

- 后台：`start_first_segment_thread` → `execute_basic_pipeline`。
- Smart Planner 拆解为 `chat` | `search` | `agent` 子步顺序执行。
- 可选 `workspace_refs`：进入本轮上下文供规划参考；**不会**仅凭该字段单独启动工作区。

### 3.3 工作区子链（`WorkspaceAgentRun`）

| 用户操作 | 触发 |
|----------|------|
| 无单独「发消息就启动工作区」的 HTTP | 仅当 **3.2** 已进入 Basic，且 Smart Planner 排了 **`agent` 子步** 时，在该步内部同步 `run_workspace_delegate` |

- 工作区磁盘变化：轮询父 Basic 的 `GET /tasks/<id>/status/` 上 `workspace_fs_generation`，再调 `GET /api/workspace/files` 刷新树。

### 3.4 续跑与修订（共用入口，按 id 分流）

| 用户操作 | 行为 |
|----------|------|
| `pending_action` 等批准后继续 | `POST /tasks/<id>/actions/`、`POST /tasks/<id>/intervention/` 等 → `execute_after_approve`：id 为 Basic → `execute_basic_pipeline`；id 为深度 → `execute_deep_research_pipeline` |
| 深度研究「修订后继续」 | **仅** `AgentTask` → `start_after_revise_thread` → `execute_after_revise` |

## 4. 能力与设计差异

| 维度 | 深度研究（`AgentTask`） | Basic（`BasicOrchestratorRun`） | Workspace（`WorkspaceAgentRun`） |
|------|-------------------------|-----------------------------------|----------------------------------|
| 阶段模型 | 固定六阶段：plan → decide → search → read → reflect → write | Smart Planner 动态子步：chat / search / agent | 规划—工具—观测循环（`agent_loop`） |
| 规划方式 | 内置多方案 plan + decide | `smart_planner.detect_smart_plan` | 每轮 LLM 产出 tool_calls |
| 检索 | 流水线内 `_search_context` + search 阶段 LLM | `search` 子步 + 可写论文展示区 | 工作区工具（`ls` / `read` / `write` 等） |
| 选定文献 | `selected_papers` 拼入 query（展示区条目） | 检索子步自动合并引用到展示区 | 不直接消费 `selected_papers` |
| 本地命令/文件 | orchestrator 内 `local_command` / `local_file_action`，可 `pending_action` | 不走深度研究那套本地工具链 | 工作区工具，非深度研究本地链 |
| 最终产物 | Markdown 报告 + `[[RA_REPORT]]` 会话消息 | 助手回复 / 检索结果 / agent 委托摘要 | 工作区执行摘要写入会话（用户可见） |
| 与另一套 DR API | 无关（本模块 `/api/research-agent/`） | 无关 | 无关 |

---

## 5. 并发与互斥

- `_active_task(session)` 在 **Basic** 与 **深度** 的 `pending` / `running` / `pending_action` 任务间取最新一条作为「进行中」。
- 同会话再发 **3.1** 或 **3.2** 任一入口 → **409**。
- Workspace 子运行挂在 Basic 之下，不单独参与「用户再开一条 HTTP 任务」的互斥判断（父 Basic 已占位）。

---

## 6. 代码锚点

| 编排器 | 锚点 |
|--------|------|
| 深度研究 | `orchestrator.py` → `execute_deep_research_pipeline`、`PIPELINE_PHASES` |
| Basic | `basic_orchestrator.py` 文件头 docstring → `execute_basic_pipeline` |
| Workspace 委托 | `agent_orchestrator.py` → `run_workspace_delegate` |
| Workspace 管线 | `workspace_pipeline.py` 文件头 docstring → `execute_workspace_pipeline` |
| 运行类型解析 | `run_registry.py` → `run_kind`（`deep_research` \| `basic` \| `workspace`） |
| HTTP 与任务 JSON | `views.py` → `create_deep_research_task`、`create_task`、`_task_to_json` |
