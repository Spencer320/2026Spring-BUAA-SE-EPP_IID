## 2. 当前实现与需求对齐

### 2.1 与 `tasks/3.6.12访问频率与并发限制.md` 对齐

| 任务要求 | 当前代码实现 | 对齐结论 |
|---|---|---|
| 管理员设置全局频率阈值 | `AccessFrequencyRule(feature/window/max_count/is_enabled)` + 前端规则管理页 | 已对齐 |
| 管理员设置用户差异化阈值 | `UserAccessQuotaOverride(user/feature/max_count)` + 前端“用户特殊配额” | 已对齐 |
| 设置并发阈值（全局/用户） | 当前无并发阈值模型、无并发阈值管理接口、无并发限流器 | 未对齐 |
| 实时监控请求量与运行态 | 频次统计页（today total/allowed/rejected）+ DR 统计页（running/queued） | 部分对齐 |
| 自动限流保护（超限自动排队/拒绝） | `check_rate_limit` 工具函数已实现，但当前 `user_create_task` 仍为 TODO，未实际接入用户任务入口 | 部分对齐（逻辑存在，执行链路未闭环） |
| 触发预警并支持管理员介入 | DR 页面对 `violation_pending` 有前端通知，支持 `force-stop`；未形成“频次/并发触发 -> 预警 -> 策略调整”自动闭环 | 部分对齐 |

### 2.2 与 `docs/ResearchAgent` 约束关系

- `科研助手-执行引擎架构与数据模型API契约草案.md` 要求 `TaskIngress` 在任务创建时做限频校验；当前代码仅保留注释约束，尚未在用户入口执行。
- `科研助手-联网搜索与本地命令执行安全基线.md` 要求单任务并发阈值；当前仓库尚未实现并发阈值模型与强制拦截路径。
- `科研助手-SRS映射与范围定义.md` 明确管理端非首批验收范围；当前实现属于“先落地管理端基础能力，用户执行链路待补齐”状态。

## 3. 管理端页面设计（当前代码）

### 3.1 路由与菜单

- 路由：`/access-frequency`（子路由：`rules/overrides/stats`）
- 菜单：`LayoutContainer.vue` 左侧“频次控制”
- 并发可观测入口：`/deep-research/tasks`（展示 `running_count`、`queued_count`，支持人工中断）

### 3.2 页面结构

```text
AccessFrequencyManage.vue
├── FrequencyRules.vue
│   ├── 规则列表（feature/window/max_count/is_enabled）
│   └── 新增/编辑规则弹窗
├── UserOverrides.vue
│   ├── 用户覆盖列表（user/feature/max_count/reason）
│   └── 新增覆盖弹窗（remote 用户搜索）
└── FrequencyStats.vue
    ├── 今日调用概览卡片（按 feature 拆分）
    ├── 用户访问排行
    └── 用户配额详情弹窗
```

并发相关页面能力（当前）：
- `DRTaskList.vue` 提供运行中/排队中计数、任务列表、自动刷新、`force-stop` 人工干预；
- 暂无“并发阈值配置页”。

### 3.3 前端参数映射（当前代码）

- 规则接口：`feature/window/max_count/is_enabled/description` 直接透传；
- 覆盖接口：`user_id/feature/max_count/reason` 透传；
- 统计排行：`feature/date/top_n` 透传；
- DR 任务列表：`status/user_id|username/date_from/date_to/keyword/page_num/page_size`。

## 4. 后端联动与接口约束

### 4.1 已接入接口（访问频率）

- `GET /api/manage/access-frequency/rules`
- `POST /api/manage/access-frequency/rules`
- `PUT /api/manage/access-frequency/rules/{rule_id}`
- `DELETE /api/manage/access-frequency/rules/{rule_id}`
- `GET /api/manage/access-frequency/user-overrides`
- `POST /api/manage/access-frequency/user-overrides`
- `DELETE /api/manage/access-frequency/user-overrides/{override_id}`
- `GET /api/manage/access-frequency/stats`
- `GET /api/manage/access-frequency/stats/users`
- `GET /api/manage/access-frequency/stats/users/{user_id}`

### 4.2 频次执行逻辑（当前代码事实）

`business/utils/rate_limit.py::check_rate_limit` 已实现：
1. 优先查 `UserAccessQuotaOverride`；
2. 其次查启用态 `AccessFrequencyRule`；
3. `max_count=-1` 放行、`0` 拒绝；
4. 统计当前窗口已放行次数并判定超限；
5. 异步写入 `FeatureAccessLog(allowed/rejected)`。

当前限制：
- 仓库内未找到实际调用 `check_rate_limit` 的业务入口；
- `business/api/deep_research.py::user_create_task` 仍返回 TODO 占位错误，未接入限频校验。

### 4.3 并发相关后端能力（当前代码事实）

已具备：
- `GET /api/manage/deep-research/stats`：返回 `running_count/queued_count` 等并发态观测指标；
- `POST /api/manage/deep-research/tasks/{task_id}/force-stop`：管理员手动中断任务。

未具备：
- 并发阈值配置模型（如全局并发上限、用户并发上限）；
- 并发阈值管理接口；
- 并发限流自动拦截/排队器。

## 6. 后续需要完善的代码工作

### P0（优先补齐闭环）

1. 接入真实限频入口  
- 在用户任务创建链路调用 `check_rate_limit`，至少覆盖 `deep_research` 高成本入口。

2. 并发阈值能力落地  
- 新增并发阈值配置模型与管理接口（全局 + 用户级）；
- 在任务创建/调度处执行并发阈值检查，形成自动排队或拒绝策略。

3. 前后端字段口径修正  
- `GET /stats/users` 返回字段与前端排行列口径统一（当前前端使用 `total/rejected`，后端返回 `count`）；
- 覆盖列表筛选参数统一（前端传 `keyword`，后端当前按 `user_id/feature` 过滤）。

### P1（增强治理）

1. 频次与并发告警联动  
- 命中阈值时写统一告警事件，管理端支持告警列表与处置流转。

2. 站点维度限频  
- 补充“用户-站点”或“平台-站点”频控窗口，降低单域名触发风控概率。

3. 自动降级策略  
- 阈值接近时自动降速、延迟调度或切换替代来源，而非仅硬拒绝。

### P2（长期演进）

1. 分布式计数与并发控制  
- 使用 Redis 原子计数/令牌桶，避免多进程下计数偏差。

2. 策略版本化与灰度  
- 支持限频/并发策略灰度发布与回滚，降低误配置风险。

## 7. 问题与处理记录（2026-04-27）

### 7.1 问题 A：限频逻辑存在但未接入用户主链路

- 现象：管理端可配规则，但用户任务入口未触发限频判定。
- 根因：`user_create_task` 当前为 TODO 占位实现。
- 影响：`FeatureAccessLog` 无法形成真实业务流量统计闭环。

### 7.2 问题 B：用户覆盖筛选参数前后端不一致

- 现象：前端“用户名搜索”传 `keyword`，后端 GET 覆盖列表未消费该参数。
- 影响：页面搜索行为与用户预期不一致。

### 7.3 问题 C：排行字段口径不一致

- 现象：前端排行表使用 `total/rejected`，后端当前返回 `count`。
- 影响：排行表部分列展示异常或空值。

### 7.4 问题 D：并发阈值能力尚未实现

- 现象：当前仅能看 `running/queued`，并发治理依赖管理员人工中断。
- 影响：无法满足 FR-SJGL-0012“自动并发限制与保护”要求。

## 8. 全流程设计（当前代码口径）

### 8.1 已落地流程（频次配置与观测）

```text
管理员配置规则（Rules/Overrides）
  -> 写入 AccessFrequencyRule / UserAccessQuotaOverride
  -> 管理端查看 today/ranking/detail 统计
  -> 结合 DR 监控页观察 running/queued 并人工中断异常任务
```

### 8.2 待补齐流程（自动限频与并发限制）

```text
用户创建任务
  -> TaskIngress 调用 check_rate_limit（当前待接入）
  -> 命中频次阈值：拒绝并写 rejected 日志
  -> 未命中：放行并写 allowed 日志
  -> 并发阈值校验（当前待实现）：
       超限 -> 入队/拒绝 + 告警
       未超限 -> 正常调度执行
  -> 管理端统一展示：频次统计 + 并发态 + 告警处置
```

### 8.3 与 `docs/ResearchAgent` 的一致性要求

- 保持“默认拒绝、可审计、可解释”原则：拒绝与降级都必须产生日志与可读原因；
- 管理端只做治理与可视化，不替代用户主链路中的执行拦截；
- 频次与并发治理需要与执行引擎入口绑定，才能形成真正闭环。
