## 1. 现有项目结构概览

```text
EPP-Frontend-Manager-Dev/
└── src/
    ├── api/
    │   └── deep_research.js
    ├── router/
    │   └── index.js
    ├── views/
    │   ├── layout/
    │   │   └── LayoutContainer.vue
    │   └── deep_research/
    │       ├── DRMonitorManage.vue
    │       ├── DRTaskList.vue
    │       ├── DRTaskDetailDrawer.vue
    │       ├── DRTraceDrawer.vue
    │       └── DRAuditLogs.vue
```

**技术栈**：Vue 3（Options API）+ Vue Router 4 + Element Plus + Axios + Sass  
**现状能力**：已具备 DR 任务监控、强制中断、屏蔽输出、审计日志展示能力。  
**本次目标**：在此基础上形成“合规审计”独立入口与专项视图，强化违规预警与处置闭环。

### 2.2 设计原则

- 与现有 DR 监控模块解耦：保留原 `/deep-research` 监控入口，新增 `/deep-research-compliance` 审计入口。
- 合规优先视图：在合规审计页默认聚焦 `violation_pending`、`needs_review` 两类任务。
- 降低学习成本：复用既有表格、抽屉、日志页面，避免平行造轮子。

## 3. 功能目标与范围

### 3.1 目标

围绕 FR-SJGL-0008，提供以下管理端能力：

1. 实时识别“违规待处理”任务并高亮展示。  
2. 管理员可快速执行“强制中断 / 强制屏蔽 / 恢复输出”。  
3. 审计日志可按管理员、操作类型、时间范围检索。  
4. 违规任务与审计日志形成可追溯闭环。

### 3.2 范围边界

- **在范围内**：管理端路由、菜单、列表筛选、日志筛选、预警提示、处置入口。
- **不在范围内**：用户端报告页面重构、算法策略调优、外部审核 API 供应商替换。

### 3.3 需求条目对齐（3.6.8）

| 需求条目 | 设计落点 |
|---|---|
| 3.6.8.1 简述 | 新增管理端并列入口 `DR 合规审计`，建立“违规发现-人工干预-用户侧屏蔽提示”闭环。 |
| 3.6.8.2 前置条件 | 依赖 DR 任务监控链路可用；依赖内容安全审核 API 已配置并可返回命中结果。 |
| 3.6.8.3 主要流程 | 对应本方案的任务列表实时扫描标记、合规面板预警、强制中断、强制屏蔽与审计日志追踪。 |
| 3.6.8.4 效果 | 通过过程+结果双扫描与人工处置入口，降低敏感内容泄露与违规报告外显风险。 |

**流程映射说明**：
1. 系统在“规划-检索-阅读-反思-成文”阶段产生的数据命中红线后，任务进入 `violation_pending`。  
2. 合规审计页优先展示 `violation_pending` / `needs_review`，并触发前端预警通知。  
3. 管理员可在列表或详情抽屉执行 `force-stop`。  
4. 管理员可执行 `suppress-output`，用户端统一展示“内容违规，无法展示”。

## 6. API 模块设计

沿用 `src/api/deep_research.js`，不新增独立 API 文件。

### 6.1 已复用接口

| 方法 | 路径 | 用途 |
|---|---|---|
| GET | `/api/manage/deep-research/stats` | 统计卡片 |
| GET | `/api/manage/deep-research/tasks` | 任务列表（含筛选） |
| POST | `/api/manage/deep-research/tasks/{id}/force-stop` | 强制中断 |
| POST | `/api/manage/deep-research/tasks/{id}/suppress-output` | 强制屏蔽 |
| POST | `/api/manage/deep-research/tasks/{id}/unsuppress-output` | 恢复输出 |
| GET | `/api/manage/deep-research/audit-logs` | 全局审计日志 |
| GET | `/api/manage/deep-research/tasks/{id}/audit-logs` | 任务审计日志 |

### 6.2 新增前端传参与兼容

- 任务列表接口支持：
  - `username`（用户名模糊匹配）
  - `keyword`（研究问题关键词）
  - `status`（多值）
- 审计日志接口支持：
  - `admin_id`（管理员 ID）
  - `admin_name`（管理员名称模糊匹配）
  - `action`、`date_from`、`date_to`

## 7. 视图组件设计

### 7.1 `DRComplianceAudit.vue`（新）

**职责**：合规审计总入口，组织“违规任务监控 + 审计日志”双标签。  
**结构**：

```text
DRComplianceAudit.vue
├── el-alert（模块说明）
└── el-tabs
    ├── 违规任务监控 -> DRTaskList(fixedStatusList=['violation_pending','needs_review'])
    └── 审计日志 -> DRAuditLogs
```

### 7.2 `DRTaskList.vue`（增强）

**新增能力**：
- 支持“固定状态视图模式”（用于合规页仅看违规任务）。
- 筛选项增强：
  - 用户关键词（用户名/用户ID）
  - 研究问题关键词
- 新增风险列：
  - `violation_pending` 显示 `待处理`
  - `violation_count > 0` 显示 `命中 N`
- 新增预警通知：
  - 首次看到 `violation_pending` 任务触发 `ElNotification`。
- 操作按钮放宽到合规状态：
  - `force-stop`：`running/queued/violation_pending/needs_review`
  - `suppress-output`：`completed/violation_pending/needs_review` 且未屏蔽

### 7.3 `DRAuditLogs.vue`（增强）

**新增能力**：
- 管理员筛选支持“姓名/ID 二选一输入”。
- 操作类型新增 `auto_violation`（自动违规命中）。
- 列表字段保持稳定：管理员、任务ID、动作、原因、时间。

### 7.4 `DRTaskDetailDrawer.vue`（增强）

**新增能力**：
- 审计动作映射新增 `auto_violation`。
- 详情抽屉内操作权限与任务列表保持一致（支持违规状态中断/屏蔽）。

## 8. 后端联动与数据设计

> 本节用于前端联调约束，便于与后端并行开发。

### 8.1 数据模型增强（DR 任务）

`DeepResearchTask` 新增：

- `violation_count`
- `latest_violation_reason`
- `latest_violation_phase`
- `latest_violation_source`
- `latest_violation_at`
- `report_compliance_hash`

`DeepResearchAuditLog.action` 新增：

- `auto_violation`

### 8.2 自动审计链路

- `DeepResearchStep` 创建后触发步骤文本扫描。
- `DeepResearchTask.report` 更新后触发报告扫描（带 hash 去重）。
- 命中违规后：
  - 任务状态置为 `violation_pending`（终态任务除外）
  - 写入 `auto_violation` 审计日志

### 8.3 合规处置联动

- 管理员屏蔽输出后：
  - 若任务状态为 `violation_pending`，转 `needs_review`
  - 用户端统一提示：`内容违规，无法展示`
  - 向用户发送站内通知
