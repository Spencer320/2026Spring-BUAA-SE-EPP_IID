# 科研助手模块：论文展示区与工作区

**新机制「论文展示区」**、**工作区交互与工具**。

---

## 1. 论文展示区（Paper shelf）

### 1.1 数据模型：`ResearchPaperShelfItem`

- 每条记录属于一个 `ResearchSession`。  
- **`source_kind`**：  
  - `external_link`：检索得到的外链文献；  
  - `workspace_file`：用户从个人工作区选中的文件。  
- **`dedupe_key`**：同会话唯一（外链一般为规范化 URL，工作区为路径派生键），用于去重。  
- **`context_tier`**：粗粒度上下文档位（如 `abstract_only`、`link_only`、`full_text_available`、`workspace_opaque`），供后续深度研究或前端展示策略参考。  
- 其它字段：`display_title`、`authors`、`abstract`、`primary_url`、`workspace_rel_path`、`file_extension`、`added_via`、`search_query`、`source_detail`（JSON）等。

### 1.2 数据从哪来？

1. **检索自动写入**：Basic 编排器执行 `search` 子步时，调用 `append_search_citations_to_shelf`（`paper_shelf.py`），把检索引用合并进展示区（按 `dedupe_key` 去重）。  
2. **用户手动添加工作区文件**：`POST /api/research-agent/sessions/<session_id>/paper-shelf/workspace/`，Body 需 `workspace_rel_path`（或 `rel_path`）；服务端用 `safe_resolve` 校验路径必须在用户工作区内且为文件。  
3. **删除**：`DELETE .../paper-shelf/<item_id>/`。

### 1.3 与深度研究的衔接

- 独立深度研究 API 的 `selected_papers` **只能**引用**当前会话**展示区已有条目的 id（`_validate_selected_papers_from_shelf`）。  
- 规划阶段会把选定文献摘要拼进用户侧上下文字符串（`orchestrator._deep_research_augment_user_query`），供 LLM 消费。

### 1.4 前端列表 API

- `GET /api/research-agent/sessions/<session_id>/paper-shelf/`  
- 单条结构由 `shelf_item_to_api_dict` 序列化（见 `paper_shelf.py` / `views.paper_shelf_list`）。

## 2. 工作区：用户侧与 Agent 侧

### 2.1 用户工作区 HTTP（不在 `research_agent` 包内，但强相关）

业务路由（`backend/urls.py`）：

- `GET /api/workspace/files` — 列目录（`?path=` 子目录）  
- `POST /api/workspace/files` — 上传  
- `GET/DELETE /api/workspace/files/<rel_path>` — 下载 / 删除  

科研助手消息里可带 `workspace_refs`（见 `session_context`），由后端校验后写入消息 `metadata` 与本轮 `BasicOrchestratorRun.runtime_config`，供 Smart Planner / 子步使用「用户点了哪些文件」这一事实。

### 2.2 Agent 自动操作工作区（Workspace pipeline）

- **触发方式**：仅当 Basic 编排的 Smart Plan 出现 **`agent`** 子任务时，`basic_orchestrator` 调用 `agent_orchestrator.run_workspace_delegate`。  
- **执行器**：`workspace_pipeline.execute_workspace_pipeline`，绑定 `WorkspaceAgentRun`。  
- **循环**：多轮 LLM JSON（`finished` / `assistant_message` / `tool_calls`）→ `run_llm_workspace_tool_batch` → 写 transcript、写审计、必要时写会话助手消息。  
- **与深度研究关系**：工作区管线**不**挂在 `execute_deep_research_pipeline` 上；深度研究若需本地文件能力，走的是 orchestrator 里另一套「本地命令 / 本地文件」工具链（可能产生 `pending_action`），与 `WorkspaceAgentRun` 不是同一条线。

### 2.3 工具一览（Unix 风格短名）

定义见 `tools/workspace_agent_tools.py` 中 `format_tools_catalog_markdown` 与 `workspace_executor.SUPPORTED_ACTIONS`，主要包括：

| 动作 | 用途摘要 |
|------|----------|
| `ls` | 列目录 |
| `read` | 读文件，支持按行区间 |
| `write` | 写/追加/按行替换 |
| `mkdir` / `rm` / `cp` / `mv` | 目录与文件管理 |
| `download` | 从 URL 拉取到工作区内 |
| `find` | 按 glob 查找 |
| `grep` | 按正则检索内容 |
| `tar` / `untar` | zip 打包与解压 |
| `extract_pdf` | PDF 抽取文本 |

参数合并与安全边界在 `workspace_executor.coalesce_workspace_args` / `execute_workspace_action` 中实现；单文件大小等受 settings（如 `RA_WORKSPACE_MAX_TEXT_BYTES`）约束。

### 2.4 前端如何感知「Agent 改了磁盘」？

- 每轮工具批次持久化后，可能递增子运行 `runtime_config.workspace_fs_generation`；若存在 `parent_basic_run`，**同步**递增父 `BasicOrchestratorRun` 上的同名字段。  
- 前端轮询 **`GET /api/research-agent/tasks/<basic_task_id>/status/`**（或会话里的 `active_task`）发现整数变化后，再请求 **`GET /api/workspace/files`** 刷新文件树（注释说明 HTTP 无服务端推送）。

### 2.5 行为审计

工作区、basic、deep 三类运行写 `AgentBehaviorAuditLog` 时通过不同外键互斥关联（模型约束「三选一」），便于按会话 / 用户筛选外联访问轨迹。
