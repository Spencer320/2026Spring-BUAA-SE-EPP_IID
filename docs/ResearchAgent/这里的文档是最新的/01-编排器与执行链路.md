# 科研助手模块：编排器与执行链路（完整版）

本文说明 `research_agent` 包内**有几类编排、各自持久化实体、以及从用户请求到落库的执行链路**。代码以仓库当前实现为准。

---

## 1. 先分清三件事：运行实体、编排器、线程入口

### 1.1 三种「运行」持久化模型（UUID 各不相同）

| 模型 | 含义 | 典型 `result_payload.runtime_config` 标记 |
|------|------|---------------------------------------------|
| `AgentTask` | **独立深度研究**六阶段流水线 | `deep_research_pipeline: true`，可选 `selected_papers` |
| `BasicOrchestratorRun` | **会话主编排**（科研助手日常对话/检索/委托工作区） | Smart Planner 的 `smart_plan` 等 |
| `WorkspaceAgentRun` | **工作区子运行**（多轮工具循环），由 basic 的 `agent` 步同步委托 | `workspace_pipeline: true`，`parent_basic_run` 非空 |

多态解析见 `run_registry.py`：`run_kind()` 返回 `deep_research` | `basic` | `workspace`；`resolve_owned_run()` 按 UUID 在三张表里查找。

### 1.2 「编排器」在文档语境里指三层逻辑（不是三个 Django App）

1. **Basic 编排器** — `basic_orchestrator.execute_basic_pipeline`  
   - 会话用户消息的**唯一直接**编排入口（HTTP 创建任务后 `start_first_segment_thread` 调它）。  
   - Smart Planner 拆成有序子步骤：`chat` → `search` → `agent`（可扩展为多步）；步间缺参用 `step_refill`。  
   - **`agent` 步**内部同步调用下面第 3 层。

2. **深度研究编排（六阶段）** — `orchestrator.execute_deep_research_pipeline`  
   - **仅**操作 `AgentTask`。  
   - 语义上的六阶段常量：`PIPELINE_PHASES = ("plan", "decide", "search", "read", "reflect", "write")`（见 `orchestrator.py` 顶部）。  
   - 实现上对每个子任务会在 `search` / `read` / `reflect` 上多轮循环，并在 `steps` 里留下多段 `phase` 记录；进度条里对深度任务按「约 6 段」估算百分比。

3. **工作区子编排（Agent 委托）** — `agent_orchestrator.run_workspace_delegate` → `workspace_pipeline.execute_workspace_pipeline`  
   - **不**由普通「发一条会话消息」的 HTTP **单独**拉起；由 Basic 编排器在 `agent` 子任务里**同步**创建 `WorkspaceAgentRun` 并跑完。  
   - 与深度研究流水线**解耦**（`workspace_pipeline` 模块头注释写明不得与 `execute_deep_research_pipeline` 混为一条链）。

---

## 2. 执行链路（按用户场景）

### 2.1 科研助手（会话）：`BasicOrchestratorRun`

1. 用户调用 `POST .../tasks/`、`POST .../sessions/messages/` 等（见 API 文档），写入用户消息 + 占位助手消息，创建 `BasicOrchestratorRun`，`status=pending`。  
2. `orchestrator.start_first_segment_thread(run_id)` →（非 sqlite）后台线程 → `execute_basic_pipeline(run_id)`。  
3. `execute_basic_pipeline`：  
   - 规划 / 补参 / 顺序执行 `chat`、`search`、`search` 可把检索引用写入论文展示区（`append_search_citations_to_shelf`）。  
   - 遇到 `agent`：`run_workspace_delegate(basic_run_id, ...)` → 新建 `WorkspaceAgentRun` → `execute_workspace_pipeline`（多轮 LLM + 工作区工具）→ 返回文本摘要写回 basic 的链式上下文。  
4. 任务结束：`BasicOrchestratorRun` 终态；会话消息列表由各步写入。

### 2.2 深度研究（独立入口）：`AgentTask`

1. 用户调用 `POST .../tasks/deep-research/`，创建 `AgentTask`，`runtime_config` 含 `deep_research_pipeline` 与规范化后的 `selected_papers`（可选）。  
2. `start_deep_research_thread(task_id)` → `execute_deep_research_pipeline(task_id)`。  
3. 流水线：`plan` → `decide` → 对每个子任务循环：`search`（含检索与可选本地命令/文件干预）→ `read` → `reflect`，满足条件后进入最终 `write`，产出报告等写入 `result_payload`。  
4. 与 Basic 编排**无互相调用**；同一会话若已有 `pending`/`running`/`pending_action` 的另一类运行，HTTP 层会 409（见 `views._active_task`）。

### 2.3 工作区子运行：`WorkspaceAgentRun`

- 唯一规范入口：`run_workspace_delegate`（由 basic 的 `agent` 步调用）。  
- `execute_workspace_pipeline`：bootstrap → agent_loop（每轮规划 JSON：`finished` / `assistant_message` / `tool_calls`）→ finalize；工具见 `tools/workspace_agent_tools.py` 与 `workspace_executor.py`。  
- 文件树刷新：每轮工具批次后可 bump `runtime_config.workspace_fs_generation`；若存在 `parent_basic_run`，会同步 bump 父 Basic 上的同字段，便于前端轮询父任务 `.../status/` 后去拉 `GET /api/workspace/files`。

### 2.4 其他线程入口（与「首段」并行存在）

- `start_after_approve_thread`：`execute_after_approve` — 若 id 对应 `BasicOrchestratorRun` 则跑 `execute_basic_pipeline`，若对应 `AgentTask` 则跑 `execute_deep_research_pipeline`（用于人机协同批准后续跑）。  
- `start_after_revise_thread`：**仅**深度研究 `AgentTask` 的「修订后继续」。

## 3. 与旧测试/认知对齐的要点

- **`AgentTask` 不再承载 basic/workspace 状态**（迁移 `0007_split_orchestration_runs` 的设计）。  
- 单测若仍假设「一个 Task 模型走天下」，需要按三表拆分更新。  
- SQLite 下部分线程入口改为**同步**执行（见 `start_first_segment_thread` 等分支），集成测试需注意。
