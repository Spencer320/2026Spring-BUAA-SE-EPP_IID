# 科研助手模块：编排器与执行链路

---

## 1. 编排器

| 模型 | 含义 | 位置 |
|------|------|---------------------------------------------|
| `AgentTask` | **独立深度研究**六阶段流水线 | `pipelines.deep.orchestrator.execute_deep_research_pipeline` |
| `BasicOrchestratorRun` | **会话主编排**（科研助手日常对话/检索/委托工作区） | `pipelines.basic.orchestrator.execute_basic_pipeline` |
| `WorkspaceAgentRun` | **工作区子运行**（多轮工具循环），由 basic 的 `agent` 步同步委托 | `pipelines.workspace.agent.run_workspace_delegate` → `pipelines.workspace.pipeline.execute_workspace_pipeline` |

---

## 2. 用户入口：你怎么操作，会触发哪条链

以下路径均相对于 **`/api/research-agent/`**；完整路由与字段见 [02-前端API对接.md](./02-前端API对接.md)。

### 2.1 日常科研助手（Basic → `BasicOrchestratorRun`）

你在下面**任一入口**提交本轮问题（请求体里至少要有 `content` 或 `query`），都会创建一条 **Basic** 任务并进入 `execute_basic_pipeline`（对话 / 检索 / 按规划执行子步；`search` 可把引用写入论文展示区）：

| 用户操作 | HTTP |
|----------|------|
| 直接开任务（可顺带新建会话） | `POST /tasks/` |
| 新建会话并带第一句话 | `POST /sessions/messages/` |
| 在已有会话里发消息 | `POST /sessions/<session_id>/messages/` |
| 上一轮任务已 **completed** 后接着问（再起一条 Basic） | `POST /tasks/<父任务 id>/follow-up/` |

可选：请求体带 **`workspace_refs`**，表示用户勾选的工作区文件，会进入本轮上下文供 **Smart Planner** 参考；**不会**单独靠该字段「只启动工作区」——是否跑工作区子链见 **2.3**。

### 2.2 独立深度研究（Deep → `AgentTask`）

| 用户操作 | HTTP |
|----------|------|
| 明确走深度研究；可选用当前会话论文展示区条目 | `POST /tasks/deep-research/`（`selected_papers` 非空时必须带合法 `session_id`） |

后台进入 **`execute_deep_research_pipeline`**，与 Basic **互不调用**。若同会话里已有 `pending` / `running` / `pending_action` 的 Basic 或深度任务，再发 **2.1** 或 **2.2** 的入口会得到 **409**（`api.views._active_task`）。

### 2.3 工作区子链（Workspace → `WorkspaceAgentRun`，无单独「发消息」入口）

用户**没有**「只提交一条 HTTP 就单独启动工作区 Agent」的入口。只有当你在 **2.1** 已进入 Basic 后，本轮 **Smart Planner** 在计划里排了 **`agent` 子步**，才会在该步内部 **同步** `run_workspace_delegate` → 创建 **`WorkspaceAgentRun`** → `execute_workspace_pipeline`。

工作区磁盘若有变化：轮询父 Basic 的 **`GET /tasks/<basic_task_id>/status/`**（或会话里的 `active_task`）上的 **`workspace_fs_generation`**，变化后再调 **`GET /api/workspace/files`** 刷新文件树。

### 2.4 续跑与修订（人机协同之后）

| 用户操作 | 触发 |
|----------|------|
| 任务处于 `pending_action` 等，你在前端完成批准 / 干预后继续 | `POST /tasks/<id>/actions/`、`POST /tasks/<id>/intervention/` 等（以实现为准）→ `execute_after_approve`：id 为 Basic 则续跑 `execute_basic_pipeline`，为深度任务则续跑 `execute_deep_research_pipeline` |
| 深度研究「修订后继续」 | 仅 **`AgentTask`** 的修订后续流程 → `start_after_revise_thread` |

## 3. 与旧测试/认知对齐的要点

- **`AgentTask` 不再承载 basic/workspace 状态**（迁移 `0007_split_orchestration_runs` 的设计）。  
- 单测若仍假设「一个 Task 模型走天下」，需要按三表拆分更新。  
- SQLite 下部分线程入口改为**同步**执行（见 `start_first_segment_thread` 等分支），集成测试需注意。
