# 科研助手技术选型与工具链确认（P0）

## 1. 文档目标与边界
- 目标：明确后续实施阶段每个步骤使用的工具链，形成统一开发、测试、发布与运维口径。
- 适用范围：`P0` 用户侧与智能体侧主链路（入口 -> 引擎 -> 工具 -> 路由 -> 图文 -> 前端 -> 测试发布）。
- 非目标：管理端页面与管理端业务流程实现；仅保留后端兼容字段与接口预留。

## 2. 选型原则
- 与现有仓库一致：优先复用当前后端、前端与测试基础设施。
- 可审计优先：所有关键动作需有结构化事件与追踪能力。
- 安全可控：联网与本地执行默认最小权限、白名单、可回放。
- 低迁移成本：尽量避免引入高侵入性新基础设施。
- 可灰度发布：支持功能开关、降级、回滚。

## 3. 关键前置决策（开发前必须冻结）
- 决策 1：不依赖“模型自带联网”
  - 工程口径统一为“模型默认不具备联网能力”，即使某些供应商模型支持联网，也不作为主链路能力。
  - 所有外部数据获取必须经过 `WebSearchExecutor` / `WebFetchExecutor`，由平台侧统一做白名单、超时、审计。
- 决策 2：统一采用“类 OpenClaw”执行协议
  - 协议固定为：`感知(Perception) -> 规划(Planning) -> 执行(Action) -> 观察(Observation) -> 纠偏(Correction)`。
  - 该协议由 `StateMachine + ToolRouter + Observer + DecisionLoop` 共同实现，不是单一模块。
- 决策 3：本地操作不通过人工终端
  - 本地命令由后端 `LocalCommandExecutor` 在受控子进程中执行。
  - 禁止“人工在 IDE/服务器开终端代执行”进入产品主流程。
- 决策 4：工具调用必须结构化
  - 模型只产出工具意图（tool name + arguments），不直接产出最终事实结论。
  - 所有工具结果需转换为统一 schema，再返回模型做阅读与归纳。
- 决策 5：高风险动作必须可中断
  - 命中高风险策略时进入 `pending_action`，等待用户 `allow/revise/abort`，未确认不可继续。

## 4. 基线技术栈确认

### 4.1 后端与运行时
- 语言/框架：`Python 3.x + Django`（沿用现有服务架构）。
- API 风格：`REST + JSON`，统一 `ok/fail` 响应格式。
- 并发与任务执行：`Django 进程内编排线程（P0）`，后续可平滑迁移队列化执行。
- 配置管理：环境变量 + 项目配置文件（模型路由、阈值、白名单）。

### 4.2 数据与持久化
- 主存储：沿用当前关系型数据库（Django ORM 管理）。
- 核心模型：`DeepResearchTask`、`DeepResearchStep`、`ToolAuditEvent`（或同等审计表）。
- JSON 结构存储：用于报告、引用列表、动作卡片、审计扩展字段。

### 4.3 前端
- 框架：沿用 `Vue` 前端工程（现有 `ResearchAgent` 页面体系）。
- 交互机制：轮询优先（P0），后续可升级为流式推送。
- 展示能力：任务进度、步骤流、人工干预卡片、图文报告与导出入口。

### 4.4 质量与工程
- 单元/集成测试：沿用现有 Python 测试框架与项目测试目录规范。
- 接口调试：项目内 API 测试脚本 + 手工联调（按验收清单）。
- 代码规范：沿用现有仓库 lint/格式化标准。

## 5. 工具执行架构（回答“具体怎么操作”）

### 5.1 联网搜索执行器（`WebSearchExecutor`）
- 输入：`{query, intent, constraints}`。
- 执行流程：
  1. 查询改写与来源选择（模型仅给出意图，不直接联网）；
  2. 由执行器调用外部搜索/抓取适配器；
  3. 统一结构化为 `query/source/url/published_at/snippet/confidence`；
  4. 写入 `ToolAuditEvent` 后返回编排引擎。
- 安全控制：域名白名单、请求超时、重试退避、输出截断、可疑来源降权。

### 5.2 本地命令执行器（`LocalCommandExecutor`）
- 输入：`{template_id, args}`，参数需通过 schema 校验。
- 执行流程：
  1. 根据 `template_id` 解析预定义命令模板；
  2. 在任务隔离目录创建受控子进程执行（如 `subprocess`）；
  3. 施加超时、资源配额、输出大小限制；
  4. 返回标准化结果 `{exit_code, stdout, stderr, truncated}`。
- 关键说明：这是服务端自动执行流程，不要求也不允许人工开启终端手动执行。

### 5.3 “类 OpenClaw”落地映射
- `Perception`：读取用户输入、任务上下文、历史步骤与资源预算。
- `Planning`：拆分子任务并生成动作计划。
- `Action`：通过 `ToolRouter` 调用模型、搜索、本地命令等执行器。
- `Observation`：聚合结果质量、风险命中、失败类型、成本指标。
- `Correction`：触发重试、降级、人工确认或终止，再回到下一轮 `Perception`。

## 6. 实施步骤到工具链映射（P0）

### 6.1 步骤 1：入口与任务创建
- 目标：替换 Mock 入口，完成真实任务创建。
- 工具链：
  - `Django View + 路由`：创建/查询/中止/追问/导出接口。
  - `rate_limit` 能力：限频与配额检查。
  - `Django ORM`：任务初始化落库。
- 关键约束：
  - 参数校验必选（`query`、`mode`、`image_enabled`、`risk_policy`）。
  - 创建任务前完成限频判定，失败即返回可解释错误。

### 6.2 步骤 2：编排引擎主循环
- 目标：落地 `规划 -> 检索 -> 阅读 -> 反思 -> 写作` 状态机。
- 工具链：
  - `orchestrator` 主循环模块（Python）。
  - `StateMachine` 抽象（阶段迁移与终态收敛）。
  - `Observer + DecisionLoop`（重试/降级/人工确认决策）。
  - `DeepResearchStep` 步骤日志落库。
- 关键约束：
  - 每阶段写入状态、摘要、token、耗时、引用。
  - 异常统一收敛为 `failed` 并附错误码/错误摘要。
  - 每轮循环必须体现 `Perception -> Planning -> Action -> Observation -> Correction`。
  - `reflect` 阶段输出必须为严格 JSON，包含 `needs_optimization(yes|no)` 与 `suggestions`；
  - 当 `needs_optimization=yes` 时，`suggestions` 必须作为下一轮输入的一部分回灌 `planning/searching/reading`，避免“假优化”；
  - 必须配置并执行 `max_reflect_rounds` 上限控制，超过上限强制收敛到终态。

### 6.3 步骤 3：工具层实现（联网 + 本地）
- 目标：接入受控工具执行。
- 工具链：
  - 联网搜索执行器：白名单域名 + 超时 + 退避重试 + 结构化回包。
  - 本地命令执行器：模板白名单 + 参数 schema 校验 + 隔离目录子进程执行。
  - 工具路由器：按任务类型和风险等级派发工具调用。
  - 审计事件写入：记录 `tool_type/action/risk/status` 等字段。
- 关键约束：
  - 模型不可直接访问公网；必须通过搜索执行器取数。
  - 非白名单域名或命令必须拒绝。
  - 高风险动作进入 `pending_action`，等待用户确认。

### 6.4 步骤 4：模型路由实现
- 目标：按文档中分层策略接入模型与回退。
- 工具链：
  - 模型适配层（统一请求/响应协议）。
  - 路由配置中心（主备模型优先级、阈值、预算）。
  - 输出校验器（结构化 schema 校验，失败触发回退）。
  - 重排组件：`BGE-Reranker-V2-M3`。
- 关键约束：
  - 主模型不可用时自动降级，不可无限重试。
  - 关键产出必须可结构化校验，避免脏输出进入报告。
  - 模型输出中的“联网建议”仅作为工具调用意图，不视为已取证事实。

### 6.5 步骤 5：图文输出实现
- 目标：报告主链路 + 条件图像生成。
- 工具链：
  - 报告构建器：`Markdown + JSON` 双格式输出。
  - 图像生成调用：`Qwen_Image`（仅满足触发条件时）。
  - 引用绑定器：文本结论与图像产物共享引用链。
- 关键约束：
  - 默认仅文本；用户显式请求且安全判定通过才触发图像生成。
  - 图像生成事件与报告事件统一审计。

### 6.6 步骤 6：前端联调
- 目标：用户侧可见全流程与干预能力。
- 工具链：
  - 任务状态轮询接口：`status/events`。
  - 干预动作接口：`allow/revise/abort`。
  - 报告展示组件：正文、引用、附件（图片）与下载入口。
- 关键约束：
  - 前端展示状态必须与后端真实状态一致。
  - 干预动作需有明确反馈并驱动状态变化。
  - 对 `pending_action` 必须展示风险原因与可选动作，不允许静默阻塞。

### 6.7 步骤 7：测试与发布
- 目标：确保可用、可控、可回滚。
- 工具链：
  - 单元测试：状态机、路由器、工具执行器、触发规则。
  - 集成测试：创建任务到报告输出全链路。
  - 端到端联调脚本：前后端关键路径 + 异常路径。
  - 灰度策略：功能开关、指标监控、快速回滚。
- 关键约束：
  - 覆盖超时、域名阻断、命令拒绝、人工中断等异常场景。
  - 灰度期间重点观察失败率、平均耗时、人工干预率。

## 7. 每步骤“需彻底明确”的技术问题检查表
- 步骤 1：任务创建是否已明确风险策略、图片开关、预算字段默认值。
- 步骤 2：状态机是否已定义挂起态、终态、恢复策略与最大重试。
- 步骤 3：是否已明确模型不直连外网、执行器接口 schema、命令模板清单。
- 步骤 4：是否已定义主备回退顺序与结构化校验失败处理。
- 步骤 5：图像生成触发条件是否显式，是否绑定同任务引用链。
- 步骤 6：前端是否处理 `pending_action` 与错误码分支。
- 步骤 7：测试是否覆盖“工具不可用/审批超时/输出截断/熔断恢复”。

## 8. 安全与治理工具链确认
- 策略引擎：域名白名单、命令模板白名单、风险分级规则。
- 审计链路：任务事件 + 工具事件 + 干预事件统一可追踪。
- 资源治理：token 预算、并发阈值、超时阈值、熔断规则。
- 隐私保护：输入输出脱敏快照，避免敏感信息直出日志。

## 9. 配置与环境分层建议
- `dev`：开放调试日志，低配额阈值，便于快速验证。
- `staging`：接近生产配置，开启完整审计与灰度开关演练。
- `prod`：严格白名单、严格预算熔断、保守重试策略。
- 配置载体建议：
  - `model_routing.yaml`（模型路由与回退策略）
  - `tool_policies.yaml`（域名/命令白名单与风险规则）
  - `feature_flags.yaml`（图像生成、高级能力开关）
  - `command_templates.yaml`（本地命令模板与参数 schema）
  - `executor_timeouts.yaml`（联网/本地执行超时与重试配置）

## 11. 阶段性交付验收口径
- 文档层：步骤-工具链映射明确，配置项与责任边界清晰。
- 开发层：每个步骤有对应模块、接口、数据字段与错误码。
- 测试层：正常路径与异常路径均有可执行用例。
- 发布层：具备灰度、监控、熔断、回滚闭环。

## 12. 结论
- `P0` 阶段采用“复用现有栈 + 增量能力建设”方案，最小化迁移成本。
- 工具链已覆盖后续实施的每个步骤，可直接拆分为开发任务与测试任务。
- 管理端保持“接口与字段预留”策略，不阻塞用户侧主链路落地。
