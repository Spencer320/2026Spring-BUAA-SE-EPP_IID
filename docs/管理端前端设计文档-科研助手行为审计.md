## 3. 管理端页面设计（当前代码）

### 3.1 路由与菜单

- 路由：`/research-agent-audit`
- 菜单：`LayoutContainer.vue` 左侧新增“科研助手行为审计”
- 导航高亮：`activeMenu` 对 `/research-agent-audit` 单独匹配

### 3.2 页面结构

```text
ResearchAgentBehaviorAudit.vue
├── 筛选栏
│   ├── userName / taskName / targetDomain
│   ├── operationType / exceptionStatus / dateRange
│   └── 查询 / 重置 / 导出结构化文档
├── 审计日志表格
│   ├── 时间、用户名、任务名、域名、操作类型、状态码
│   ├── 异常状态、异常说明
│   └── 操作列：查看链路
├── 分页器（page_num/page_size）
└── 链路抽屉
    ├── 任务元信息（task/session/user/status）
    └── 行为时间线（按发生时间升序）
```

### 3.3 前端参数映射

`ResearchAgentBehaviorAudit.vue` 到后端查询参数映射：

- `filters.userName -> user_name`
- `filters.taskName -> task_name`
- `filters.targetDomain -> target_domain`
- `filters.operationType -> operation_type`
- `filters.taskStatus -> task_status`
- `filters.exceptionStatus -> exception_status`
- `filters.dateRange[0/1] -> date_from/date_to`
- `currentPage/pageSize -> page_num/page_size`
- `sortState.prop -> sort_by`（`occurred_at/user_name/task_name/step_id`）
- `sortState.order -> sort_order`（`ascending -> asc`，`descending -> desc`）
- 兼容：后端仍接受 `user_id/task_id` 参数（用于历史调用方）。

### 3.4 导出行为

- 导出时移除分页参数，仅提交筛选参数；
- 后端返回 Markdown 文本，前端通过 `Blob + a[download]` 触发下载；
- 导出表格主字段为“用户名 / 任务名”（内部仍保留 ID 作为兼容筛选参数）；
- 默认导出文件名回退为 `research-assistant-audit.md`。

## 4. 后端联动与接口约束

### 4.1 接口清单（已接入）

- `GET /api/research-agent/manage/behavior-logs/`
- `GET /api/research-agent/manage/tasks/{task_id}/behavior-chain/`
- `POST /api/research-agent/manage/behavior-logs/export/`

其中 `GET /api/research-agent/manage/behavior-logs/` 当前支持排序参数：
- `sort_by`: `occurred_at` / `user_name` / `task_name` / `step_id`
- `sort_order`: `asc` / `desc`

### 4.2 返回结构（当前代码事实）

日志列表：

```json
{
  "total": 123,
  "page_num": 1,
  "page_size": 20,
  "items": [],
  "operation_type_options": ["http_request", "web_search"]
}
```

链路详情：

```json
{
  "task": {
    "task_id": "uuid",
    "task_name": "string",
    "session_id": "uuid",
    "user_id": "string",
    "user_name": "string",
    "status": "running|pending_action|completed|failed|cancelled",
    "step_seq": 0
  },
  "logs": []
}
```

导出：

```json
{
  "file_name": "research-assistant-audit-YYYYMMDD_HHMMSS.md",
  "content": "# 科研助手行为审计报告...",
  "count": 100
}
```

### 4.3 鉴权与跨域约束（当前实现）

`research_agent/auth.py` 当前管理员鉴权优先依赖：

1. `X-Research-User-Id` + `X-Research-Role=admin` 请求头；或
2. 可被 `research_agent` 识别的 JWT。

当前联调口径：
- 管理端请求层已对 `/api/research-agent/manage/*` 自动附加 `X-Research-*` 头；
- 后端 `CORS_ALLOW_HEADERS` 已放行 `x-research-role`、`x-research-user-id`；
- 若后端未重启或配置未生效，前端会表现为“获取行为审计日志失败”。

## 7. 问题与处理记录（2026-04-27）

### 7.1 问题 A：管理端提示“获取行为审计日志失败”

- 现象：前端页面请求失败，表格无法加载。
- 根因：前端新增 `X-Research-*` 头后触发 CORS 预检，但后端未放行对应 header。
- 处理：
  - 后端 `CORS_ALLOW_HEADERS` 增加 `x-research-role`、`x-research-user-id`；
  - 后端重启后恢复正常。

### 7.2 问题 B：管理端页面空白（无审计日志）

- 现象：接口成功返回但 `items=[]`。
- 根因：
  1. 历史数据里审计日志条目本身为 0；
  2. 默认筛选范围不包含 `failed/cancelled`，失败任务日志不可见。
- 处理：
  - 默认筛选范围扩展为 `pending/running/pending_action/completed/failed/cancelled`；
  - 失败任务缺日志时可按运维脚本补最小 `pipeline_error` 审计记录（仅用于可观测性兜底）。

### 7.3 问题 C：任务频繁失败导致“无真实流程日志”

- 现象：任务快速进入 `failed`，日志链路很短或为空。
- 根因：`RA_LLM_API_KEY` 无效（401 authentication fails）。
- 结论：该问题与 `ResearchSession.owner_id` 字段语义或 migration merge 本身无直接冲突。
- 待办：修正并验证 LLM 配置后再观察真实审计流水生成。

## 8. 全流程设计（按当前代码）

### 8.1 审计事件采集入口

当前代码存在两条审计写入链路：

1. 编排器自动写入（主链路）
- 来源：`research_agent/orchestrator.py` 的 `_append_step -> _append_behavior_log`；
- 触发：`plan/decide/search/read/reflect/write` 阶段步骤落库时同步写入审计；
- 特点：自动补齐 `step_id`、`trace_id`，并对 `status` 做标准化映射。

2. 用户态探针上报（补充链路）
- 来源：`POST /api/research-agent/tasks/{task_id}/behavior-logs/`；
- 触发：前端或执行探针显式提交外部访问行为；
- 特点：允许上报 `request_headers/request_payload/action_payload`、`rule_hit`、`policy_version` 等细粒度字段。

### 8.2 字段标准化与入库口径

- `status` 归一化：`ok/success -> succeeded`、`error -> failed`、`pending -> pending_action`、`blocked -> rejected`；
- `rule_hit` 归一化：支持 `string/list/dict`，最终压缩为单字符串字段；
- 异常判定：`is_exception=true` 或 `response_status>=400` 时按异常路径落库；
- 主体类型：`actor_type` 非 `system/user/admin` 时回退默认值，避免脏数据扩散。

### 8.3 管理端查询与处置流程

```text
管理员进入 /research-agent-audit
  -> GET /manage/behavior-logs（默认只看 pending/running/pending_action/completed/failed/cancelled）
  -> 组合筛选 + 排序 + 分页查看
  -> 点击“查看链路” -> GET /manage/tasks/{task_id}/behavior-chain
  -> 点击“导出结构化文档” -> POST /manage/behavior-logs/export
```

导出链路补充说明：
- 导出前再次应用同一套筛选器；
- 导出上限 500 条；
- 导出内容含“筛选条件回显 + 明细表 + 分布统计（task_status / operation_type）”。

### 8.4 异常与边界处理

- `occurred_at` 非法格式：上报接口直接返回 `400`；
- `user_name` 筛选无匹配：列表直接返回空集，不报错；
- 管理端列表排序白名单：仅允许 `occurred_at/user_name/task_name/step_id`；
- 链路查询任务不存在：返回“任务不存在”错误，防止误导性空链路。

### 8.5 与 `docs/ResearchAgent` 约束对齐

- 对齐“全程审计”原则：工具调用与步骤推进都可沉淀 `task_id + step_id + trace_id`；
- 对齐“失败可解释”原则：异常状态、命中规则与异常说明在列表/导出中可追踪；
- 对齐“显式确认”原则：`pending_action` 状态在审计列表中可见，便于回放人工干预前后链路；
- 边界说明：`docs/ResearchAgent` 将管理端标注为非 P0 首批验收范围，当前实现属于“预留字段兼容基础上的提前落地页面”。
