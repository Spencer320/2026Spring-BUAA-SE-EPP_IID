## 2. 当前实现与需求对齐

### 2.1 与 `tasks/3.6.11目标站点访问管控.md` 对齐

| 任务要求 | 当前代码实现 | 对齐结论 |
|---|---|---|
| 支持白名单/黑名单策略模式 | `SiteAccessPolicyConfig.mode` + 管理端策略下拉切换 | 已对齐 |
| 支持规则维护（新增/修改/删除/启停） | 后端 `rules` 接口 + 前端规则表格与弹窗表单 | 已对齐 |
| 请求前实时判定目标站点 | `site_access_control.evaluate_target_domain()` 接入 `web_fetch/web_search/local_file` | 已对齐 |
| 命中规则后拦截并审计 | 拦截返回 `OUTBOUND_SITE_DENIED`，审计写入 `rule_hit/policy_version` | 已对齐 |
| 管理端可查看命中事件与统计 | `/events/` + `/stats/` 接口，前端事件表格与概览卡片 | 已对齐 |
| 策略版本可追踪 | 规则/策略变更自动 `bump_policy_version`，前端可见 version | 已对齐 |

### 2.2 与 ResearchAgent 安全基线约束关系

- 站点访问控制能力由“硬编码白名单”升级为“配置化策略 + 规则优先级”，符合平台化治理方向。
- 执行器保持最小权限原则：仍需通过静态 allowlist，再经过动态策略判定。
- 所有拦截可在审计日志中回溯到规则命中与策略版本，满足可追责要求。

## 3. 管理端页面设计（当前代码）

### 3.1 路由与菜单

- 路由：`/research-agent-site-access`
- 菜单：`LayoutContainer.vue` 左侧新增“目标站点访问管控”
- 导航高亮：`activeMenu` 对 `/research-agent-site-access` 单独匹配

### 3.2 页面结构

```text
ResearchAgentSiteAccessControl.vue
├── 概览卡片
│   ├── 策略模式 + policy_version
│   ├── 规则总数/启用数
│   └── 拦截数/放行数
├── 策略配置区
│   ├── mode 切换（whitelist / blacklist）
│   ├── description 说明
│   └── 保存策略 / 刷新
├── 规则维护区
│   ├── 筛选（keyword/rule_type/match_type/is_enabled）
│   ├── 规则表格（rule_id/type/match/pattern/priority/status/description）
│   └── 新增/编辑弹窗（rule_type/match_type/pattern/priority/is_enabled/description）
└── 命中事件区
    ├── 筛选（target_domain/status/tool_type）
    ├── 事件表格（occurred_at/user/task/domain/tool/status/rule_hit/policy_version）
    └── 分页（page_num/page_size）
```

### 3.3 前端参数映射

`ResearchAgentSiteAccessControl.vue` 到后端参数映射：

- 规则查询：`keyword -> keyword`、`ruleType -> rule_type`、`matchType -> match_type`、`enabled -> is_enabled`
- 规则提交：`ruleForm` 字段直接映射 `rule_type/match_type/pattern/priority/is_enabled/description`
- 事件查询：`targetDomain -> target_domain`、`status -> status`、`toolType -> tool_type`
- 分页：`eventPage/eventPageSize -> page_num/page_size`

## 4. 后端联动与接口约束

### 4.1 管理端接口清单（已接入）

- `GET /api/research-agent/manage/site-access/policy/`
- `PUT /api/research-agent/manage/site-access/policy/`
- `GET /api/research-agent/manage/site-access/rules/`
- `POST /api/research-agent/manage/site-access/rules/`
- `PUT /api/research-agent/manage/site-access/rules/{rule_id}/`
- `DELETE /api/research-agent/manage/site-access/rules/{rule_id}/`
- `GET /api/research-agent/manage/site-access/events/`
- `GET /api/research-agent/manage/site-access/stats/`

### 4.2 关键返回结构（当前代码事实）

策略查询：
```json
{
  "policy": {
    "mode": "blacklist",
    "policy_version": 3
  },
  "rule_summary": {
    "total": 10,
    "enabled": 8,
    "allow": 3,
    "deny": 7
  }
}
```

规则列表：
```json
{
  "rules": [
    {
      "rule_id": 1,
      "rule_type": "deny",
      "match_type": "suffix",
      "pattern": "example.com",
      "priority": 10,
      "is_enabled": true
    }
  ]
}
```

事件列表：
```json
{
  "items": [],
  "total": 0,
  "page_num": 1,
  "page_size": 20
}
```

统计：
```json
{
  "policy": {},
  "rules": {
    "total": 0,
    "enabled": 0,
    "allow": 0,
    "deny": 0
  },
  "events": {
    "total": 0,
    "blocked": 0,
    "allowed": 0
  },
  "top_blocked_domains": [],
  "top_allowed_domains": []
}
```

### 4.3 执行器判定链路约束

- `web_fetch_executor.allowed_get()`：
  1. 静态 allowlist 校验（`RA_ALLOWED_HOSTS`）
  2. 动态站点策略判定（`evaluate_target_domain`）
  3. 拒绝时返回 `OUTBOUND_SITE_DENIED`
- `web_search_executor`：
  - 对候选 citation 按域名执行策略过滤
  - 全部被拒绝时返回拦截错误并携带命中规则信息
- `local_file_executor`：
  - 下载目标 host 同样受站点策略约束，防止通过下载链路绕过控制

## 7. 问题与处理记录（2026-04-27）

### 7.1 问题 A：文档与代码实现对齐口径不一致
- 现象：需求文档描述为“全流程治理”，但缺少实际接口/字段清单。
- 处理：本设计文档按代码事实给出页面结构、参数映射、接口返回示例与文件落点。

### 7.2 问题 B：站点管控可能被非主链路绕过
- 风险：仅拦截 `web_fetch` 无法覆盖其它外联入口。
- 处理：已将策略判定同步接入 `web_search` 和 `local_file` 下载链路。

### 7.3 问题 C：审计可追溯性不足
- 风险：仅记录拦截状态无法定位具体规则版本。
- 处理：执行链路统一回填 `rule_hit` 与 `policy_version`，管理端事件表直接展示。

### 7.4 问题 D：前端提示“获取规则失败”且报错不可读
- 现象：管理端页面仅提示“获取规则失败”，难以定位后端根因。
- 根因：
  1. 后端 `research_agent` 迁移未执行时，`site-access` 接口会因缺表报 500；
  2. 前端错误提取仅处理 `error` 字符串，未兼容 `{ error: { code, message } }`。
- 处理：
  - 运维侧执行：`python manage.py migrate research_agent`（确保 `0005_siteaccesspolicyconfig_siteaccessrule` 已应用）；
  - 后端接口补充数据库 schema 异常兜底，统一返回 `SITE_ACCESS_SCHEMA_UNAVAILABLE`；
  - 前端统一错误提取逻辑，优先展示后端 `message/code`，并在兜底时附带 HTTP 状态码。

## 8. 全流程设计与排障清单（按当前代码）

### 8.1 策略配置与版本生效链路

```text
管理员进入站点访问管控页
  -> GET /manage/site-access/policy + /stats + /rules + /events
  -> 修改策略模式/规则
  -> PUT/POST/DELETE 对应接口
  -> 后端 bump_policy_version
  -> 新策略立即生效（当前为“保存即发布”）
```

### 8.2 运行时站点判定链路

```text
执行器发起外联前
  -> 静态 allowlist 校验（settings）
  -> evaluate_target_domain(动态策略)
  -> 命中 deny 或 whitelist miss => 拒绝
  -> 其余 => 放行并继续执行
```

覆盖入口：
- `web_fetch_executor.allowed_get`
- `web_search_executor._filter_citations_by_site_policy`
- `local_file_executor._download_file_to_dir`

### 8.3 拦截事件与审计回放链路

```text
运行时产生 site_access 决策
  -> rule_hit + policy_version 写入 AgentBehaviorAuditLog
  -> 管理端 /events 分页检索
  -> 管理端 /stats 聚合展示 blocked/allowed 与 Top Domain
```

与 `docs/ResearchAgent` 对齐点：
- 满足“全程审计、失败可解释、策略版本可追踪”。
- 仍缺“审批后发布、灰度发布、一键回滚”产品化流程。

### 8.4 联调排障最小清单（本次新增）

1. 页面报“获取规则失败”时先看 Network 中 `/api/research-agent/manage/site-access/rules/` 响应体。
2. 若返回 `SITE_ACCESS_SCHEMA_UNAVAILABLE`，先执行 `python manage.py migrate research_agent`。
3. 若返回 401/403，检查 `Authorization` 与 `X-Research-Role/X-Research-User-Id` 请求头。
4. 若返回其他 500，查看后端日志中 `site_access_views` traceback。
5. 迁移后建议最小回归：
   - `GET /policy`、`GET /rules`、`GET /stats` 均返回 200；
   - 规则新增/编辑/删除后 `policy_version` 递增；
   - 命中拦截时事件列表可见 `rule_hit` 与 `policy_version`。
